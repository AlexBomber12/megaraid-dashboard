## Problem

PR-001 through PR-003 deliver the email-alerts pipeline: transport, schema, notifier service, scheduler integration. From the operator's perspective, however, the dashboard provides **no signal** about whether alerts are actually working. If SMTP credentials silently expire, if the notifier hits a persistent error, or if a flood of events triggers throttle warnings, the operator finds out only by tailing logs or noticing alerts going missing — both of which defeat the point of out-of-band notification.

This PR surfaces alert-pipeline state in the existing overview page, so a single glance answers: "did anything notify recently? is anything pending? is anything broken?"

This is the **fourth and final** PR in the email-alerts series:
1. PR-001: SMTP transport, config, CLI. (Merged.)
2. PR-002: `notified_at` column and DAO methods. (Merged.)
3. PR-003: Notifier service plus scheduler integration. (Merged.)
4. PR-004 (this PR): UI status panel.

## Scope

This PR adds a small read-only panel to the overview page showing four pieces of information:
1. **Last alert sent at** — the most recent `events.notified_at` across the table, formatted in both UTC and Europe/Rome (CEST/CET). If no event has ever been notified, show `Never`.
2. **Pending alerts** — count of events at or above `alert_severity_threshold` that have `notified_at IS NULL` AND `occurred_at >= now - alert_suppress_window_minutes`. Aligned with the notifier's own filter.
3. **Sent in last hour** — count of events with `notified_at >= now - 1h`. Useful for spotting throttle-warning territory.
4. **Notifier health** — derived status: `optimal` if pending == 0 OR last sent < 2 minutes ago; `warning` if pending > 0 AND last sent < 10 minutes ago (notifier is working, just catching up); `critical` if pending > 0 AND last sent >= 10 minutes ago (notifier appears stuck).

In scope:
- One new section in the overview view model carrying the four pieces above.
- A new helper in the events service module to compute alert-pipeline statistics.
- A new partial template `partials/alert_status.html` rendered in `pages/overview.html`.
- Unit tests for the new helper.
- Integration tests for the route-level rendering.
- No new CSS classes (reuse existing `status-badge--*` classes from app.css).

Out of scope:
- Real-time toast notifications when a critical event is detected. The 30-second auto-refresh of the overview page handles it implicitly.
- A separate `/alerts` page. The four numbers fit in the overview without crowding.
- A test-send button in the UI. The CLI from PR-001 is sufficient for the operator and avoids the security question of "what if a malicious LAN user spams the alert inbox via the dashboard".
- Email delivery diagnostics (DKIM/SPF/DMARC self-check). Out of scope for the dashboard; operator runs the CLI test command if delivery is suspect.
- Per-recipient breakdown. There is one recipient (`alert_to`).

## Implementation notes

The `health` rule produces three states. Boundary cases worth nailing in tests:
- `pending=0` always yields `optimal` regardless of `last_alert_sent_at` (could be None, could be days ago — if nothing is pending, the notifier is not stuck).
- `pending>0` and `last_alert_sent_at is None` yields `critical` (we have pending events and have never sent anything — either the notifier never ran or has been failing since deploy).
- `pending>0` and `last_alert_sent_at < 2 minutes ago` yields `optimal` (sending in progress, normal).
- `pending>0` and `2 min <= age < 10 min` yields `warning` (catching up).
- `pending>0` and `age >= 10 min` yields `critical` (stuck).

The 2-minute and 10-minute thresholds are not configurable in this PR. They are derived from the notifier's 60-second tick — 2 minutes covers two ticks plus jitter; 10 minutes covers ten ticks of complete inaction. If those values turn out to be wrong in practice, surfacing them as config knobs is a follow-up.

The new overview ViewModel field is nested. Existing code that constructs `OverviewViewModel` instances (real or in tests) must be updated to provide the new field. Search for `OverviewViewModel(` constructor calls and update them. Default the field to a sensible empty value in test fixtures rather than making it `Optional` — explicit is better than implicit, and the production code always populates it.

The `health_label` strings are user-facing; keep them in English (the rest of the dashboard is English). Suggested: `Notifier OK`, `Notifier catching up`, `Notifier appears stuck`.

The partial template is small enough that no separate CSS file is needed. Reuse `.data-block`, `.mono`, and `.status-badge--*` from existing `app.css`. If layout adjustments are needed (e.g. four cells in a row), inline a small `<style>` block at the top of the partial scoped via a class — but only if the existing classes do not already provide the layout.

`utc_to_cest` filter is already registered. Use it directly in the template via `{{ vm.alert_status.last_alert_sent_at | utc_to_cest }}`.

The route logging line `ui_overview_rendered` does not need new fields. The alert-status section is part of the same render and adds at most 2-3 ms to the loader; it is not worth a separate timing log.
