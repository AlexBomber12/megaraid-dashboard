## Problem

PR-006/PR-007/PR-008 made the RoC temperature visible to the alerting pipeline. The operator opening the dashboard, however, still sees no number for it. Until the redesigned 6-tile status strip lands in PR-020, there must be at least one place on the overview page where the operator can see the current RoC value and its status (optimal/warning/critical) at a glance.

This is explicitly an interim PR. It adds one new tile to the existing overview status strip using current visual conventions. PR-020 will replace the entire strip; the tile added here is throwaway by design and should not invent new components or CSS structures that would have to be unwound later.

## Scope

In scope:
- Extend the overview view model with a new `RocTemperatureSection` (or equivalent) carrying current value, derived status, and the configured thresholds for tooltip/aria.
- Render the section in `templates/partials/overview_data.html` using the existing tile/card pattern from the same file.
- Empty-state behavior: when the latest snapshot has `roc_temperature_celsius is None`, render the tile with value `unknown` and a neutral badge.
- Status mapping: optimal when value below warning threshold; warning when at or above warning and below critical; critical when at or above critical. No hysteresis here (the detector handles that for events; the tile shows the latest reading directly).
- Unit tests for the view model loader and an integration test for the overview route asserting the tile renders.

Out of scope:
- Charts, trends, or sparklines for RoC. The drive-detail page already has a chart pattern; RoC trend goes in PR-052 (Grafana) or in a follow-up redesign PR.
- New CSS classes. Reuse existing `status-badge--*`, `data-block`, `mono`, and any tile classes already in `app.css`.
- Configurable display unit (Celsius only). The whole codebase is Celsius.
- Tooltip with current vs thresholds. PR-027 adds that for the redesigned tile; this interim tile shows the value and the badge.
- Replacing or rearranging the existing tiles in the strip. Append the RoC tile at the end of the strip.

## Implementation notes

This tile is interim. PR-020 will redesign the strip into 6 dedicated tiles (Controller / VD / RAID / BBU / MaxTemp / RoC). The dataclass and loader added here are the right shape to feed the redesigned RoC tile too: PR-020 can keep the same `RocTemperatureSection` and only change the template. So even though the template change is throwaway, the view-model change is durable.

The status mapping ignores hysteresis intentionally. The event detector cares about transitions; the operator looking at the dashboard cares about the current reading. If the value is 96 right now, the tile is warning right now, even if no warning event has yet been emitted. The two concerns are decoupled by design.

The "Unknown" empty state covers two distinct cases: legacy storcli without RoC support, and no snapshot at all (fresh deploy with collector still warming up). Both render the same neutral tile because, from the operator's standpoint, both translate to "no signal yet".

The label string format: when the status is `optimal` or `neutral`, show `"78 C"` or `"Unknown"`. When `warning` or `critical`, show `"96 C (warning)"` or `"107 C (critical)"`. The redundant inline status word helps screen readers and gives a plain-text description for monochrome printouts. For aria, the existing tile pattern probably already does the right thing; do not add new aria attributes unless the surrounding tiles do.

Reusing `status-badge--neutral`: if no such class exists in `app.css`, use `status-badge--info` (which is in the design tokens) or whatever neutral-toned badge the file already has. Read `app.css` once before deciding; do not introduce a new class.

The PR is scoped to around 100 lines net new. If the implementation grows past 150, something is wrong; stop, re-read the scope, and split if needed.
