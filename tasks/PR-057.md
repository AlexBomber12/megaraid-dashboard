## Problem

Two documents that pair well: RUNBOOK for "what to do when X happens" (operations playbook), ARCHITECTURE for "why the system looks this way" (one-page system overview). Both deferred per §10 because they would be wrong if written before the system was stable.

## Scope

In scope:
- `docs/RUNBOOK.md` covering routine operations:
  - Daily: check dashboard, look for new events.
  - Weekly: skim Grafana trends; review last week's events list.
  - Monthly: SQLite backup verification; quarterly security review reminder.
  - Per-incident: drive failure → replace flow link; thermal alert → check airflow / heatsink; alert flood → check notifier suppression / start maintenance mode.
  - Recovery: SQLite DB restoration; service restart; rollback to prior version.
  - Around 300 lines.
- `docs/ARCHITECTURE.md`:
  - One ASCII diagram (data flow).
  - Component descriptions: collector, detector, notifier, web, exporter, retention.
  - DB schema overview (key tables, relationships).
  - External dependencies (storcli, SMTP, nginx, Prometheus).
  - Where state lives (DB, env file, sudoers, systemd unit, journald).
  - Around 200 lines.

Out of scope:
- Code-level deep dive. ARCHITECTURE is the high-level map; for code reading, the code itself is the reference.
- Multi-host architectures.
- Tutorials. RUNBOOK is reference, not how-to.

## Implementation notes

RUNBOOK should fit the pattern: heading is the question ("Drive failed; what now?"), body is the answer with linked deeper docs.

ARCHITECTURE should NOT duplicate code. It maps; reading the code itself is the canonical truth. The architecture doc is for the reader who wants the 10-minute orientation.

Both docs are timeless if written carefully. Avoid embedding values that change (PR numbers, version strings). Reference `INSTALL.md` and `TROUBLESHOOTING.md` for time-bound details.

For incident response, the most common cases get the most detail. Likely top 3 for the homelab: drive failure (replace flow), alert flood (maintenance mode), missed alerts (notifier health check).
