## Problem

The events page currently uses a generic table layout that does not match the §3 design system, and its auto-refresh resets pagination position: when the operator scrolls into older events and the page auto-refreshes, the cursor jumps back to the most recent. This loses operator context mid-investigation.

Two fixes ship together because the redesign and the refresh fix touch the same templates and route handlers; splitting into two PRs would mean rewriting the same files twice.

## Scope

In scope:
- Replace the events table with the Timeline component pattern (per §3.3) used on the dashboard (PR-022): vertical list with severity icon at left, primary text middle, timestamp at right.
- Severity badges use the same `--state` token classes as the rest of the §3 system. Consistent across all pages.
- Pagination: cursor-based (existing). Fix the auto-refresh to NOT reset the cursor: when htmx polls for updates, the request includes the current cursor and only the most recent page is refreshed in place. New events appended at the top via `hx-swap-oob` or replace-with-merge logic.
- Filter chips at the top: severity (info/warning/critical), category (controller, vd, pd, smart_alert, temperature, controller_temperature, disk_space, ...). Multi-select; URL query reflects active filters.
- Tests: route filtering, redesigned markup, auto-refresh behavior.

Out of scope:
- Live websocket / SSE push of events. The 30 s polling is sufficient.
- Per-event detail modal. Click on a row goes to `/events/<id>` if such a route exists; if not, the row stays inline (the message field is the detail).
- Bulk actions. There are no bulk operations on events.
- Export to CSV. Future PR if needed.

## Implementation notes

The auto-refresh fix is the high-value part. The redesign is consistency. If the redesign starts ballooning, drop the filter-chips work to a follow-up MICRO PR and ship the auto-refresh fix plus timeline conversion.

The "since" mechanism vs "cursor" mechanism: cursor moves backward through history (Load more); since polls forward for new entries. Both coexist; the route distinguishes via presence of either parameter. When both are passed, since wins (poll-mode).

`hx-swap-oob` (out-of-band) is the htmx idiom for inserting fragments outside the polled element. When the route returns new events on a poll, the response markup is `<ul hx-swap-oob="afterbegin:#events-list">{{ new events }}</ul>`. htmx sees the OOB attribute and inserts the children at the top of the existing list without replacing anything.

The polling endpoint distinct from the page-fetch endpoint helps tests: route to `/events` for full page, `/events?since=N` for poll-only. The existing tests on the events route pass since they request without `since`.

The filter chips state lives in URL query params, not in browser localStorage. Operator share-link to `/events?severity=critical` lands their colleague on the same filter view.

The legacy table can be removed in this same PR; do not leave it as dead code. The redesign is the page now.
