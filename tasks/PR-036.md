## Problem

Drive replacement on a hardware RAID 6 array is a four-step physical procedure:
1. Mark the failing drive offline AND mark it missing (storcli `set offline` then `set missing`).
2. Pull the drive from the slot (physical action).
3. Insert the replacement; tell the controller to insert it in the missing slot (`add hotsparedrive` or `replace missing`).
4. Wait for rebuild to complete.

Step 1 is a pair of storcli mutations that alter array state. Done wrong, the operator can offline the WRONG drive and force a rebuild on a working drive, which on RAID 6 with one drive already faulted is a path to data loss. Done right, it is the necessary precondition for hot-swap.

This PR ships ONLY the backend for Step 1: routes, command builder, audit emission, dry-run mode. UI lands in PR-037. Steps 2-3 land in PR-038 (also high-risk). Step 4 (rebuild progress) lands in PR-039.

## Scope

In scope:
- Two new endpoints under `/drives/{e}:{s}/replace/`:
  - `POST .../offline` (sets drive offline)
  - `POST .../missing` (sets drive missing)
- Command builders in `services/drive_actions.py`:
  - `build_set_offline_command(enclosure, slot) -> list[str]`
  - `build_set_missing_command(enclosure, slot) -> list[str]`
- Storcli runner whitelist extended to accept these templates.
- Strict integer validation on path params (already covered by FastAPI int parsing).
- A state machine on the route handler: `offline` is allowed only against a drive currently in `Onln`/`Failed` state. `missing` only after `offline` succeeded. The route reads the latest snapshot to decide.
- Confirmation token: each request body includes `serial_number` of the target drive. The route verifies the serial matches the latest snapshot for that e:s slot. Mismatch returns 409 with the actual serial vs supplied. This is the §8 "destructive operations require typed confirmation of affected drive serial number" requirement.
- Dry-run mode: `?dry_run=true` runs all validation and command-building, returns the would-be argv WITHOUT invoking storcli. The runner is not called.
- Audit event on each successful command (via `record_operator_action` from PR-034).
- Tests: every state machine transition, every validation failure, command-builder output exact match, dry-run path, audit-emission, runner-whitelist accepts new templates and rejects deviations.

Out of scope:
- UI. PR-037.
- Step 2-3 commands. PR-038.
- Rebuild polling. PR-039.
- Rollback semantics. There is no in-band rollback for a set-offline; the operator must follow through with the full replace procedure or document the abort.

```python
    _validate_es(enclosure, slot)

    _validate_es(enclosure, slot)

def _validate_es(enclosure: int, slot: int) -> None:
    if not isinstance(enclosure, int) or enclosure < 0 or enclosure > 255:
        raise ValueError("enclosure must be int in [0, 255]")
    if not isinstance(slot, int) or slot < 0 or slot > 255:
        raise ValueError("slot must be int in [0, 255]")
```

```python
class ReplaceRequest(BaseModel):
    serial_number: str
    dry_run: bool = False


@router.post("/drives/{enclosure}:{slot}/replace/offline")
async def drive_replace_offline(
    enclosure: int, slot: int, body: ReplaceRequest, request: Request
) -> JSONResponse:
    return await _run_replace_step(enclosure, slot, "offline", body, request)


@router.post("/drives/{enclosure}:{slot}/replace/missing")
async def drive_replace_missing(
    enclosure: int, slot: int, body: ReplaceRequest, request: Request
) -> JSONResponse:
    return await _run_replace_step(enclosure, slot, "missing", body, request)


async def _run_replace_step(
    enclosure: int, slot: int, step: Literal["offline", "missing"],
    body: ReplaceRequest, request: Request,
) -> JSONResponse:
    with request.app.state.session_factory() as session:
        drive = _load_latest_drive(session, enclosure, slot)
    if drive is None:
        return JSONResponse({"error": "no snapshot for slot"}, status_code=404)
    if drive.serial_number != body.serial_number:
        return JSONResponse(
            {"error": "serial mismatch", "expected": drive.serial_number,
             "supplied": body.serial_number},
            status_code=409,
        )
    transition_ok = _can_transition(drive.state, step)
    if not transition_ok:
        return JSONResponse(
            {"error": f"cannot {step} drive currently in state {drive.state}"},
            status_code=409,
        )
    if step == "offline":
        argv = build_set_offline_command(enclosure, slot)
    else:
        argv = build_set_missing_command(enclosure, slot)
    if body.dry_run:
        return JSONResponse({"dry_run": True, "argv": argv})
    runner = request.app.state.storcli_runner
    result = await asyncio.to_thread(runner.run, argv)
    with request.app.state.session_factory() as session, session.begin():
        record_operator_action(
            session,
            username=request.scope.get("user_username", "unknown"),
            message=f"replace step {step} drive {enclosure}:{slot} serial {body.serial_number}",
        )
    return JSONResponse({"step": step, "argv": argv, "result": result})
```

## Implementation notes (READ BEFORE CODING)

**HAND-VERIFICATION MANDATORY before merge.** This PR controls a destructive storcli sequence on real hardware. Every storcli command emitted by this PR must be:
1. Reviewed letter-by-letter against the LSI 9270CV-8i firmware 23.34.0-0019 documentation in `docs/storcli-reference.md` (or the upstream MegaRAID storcli12 reference if the doc is not yet in repo).
2. Tested against a non-production controller OR a thoroughly mocked storcli runner returning realistic JSON, including failure responses.
3. Demonstrated to the human operator (Aleksei) before the PR is marked ready. Output of the manual test (commands sent + responses) must be pasted in the PR description.

The state machine prevents the most common operator mistake: trying to mark a drive missing while it is still online. storcli accepts this command but the result on the array is undefined; better to refuse at the application layer.

The serial-number confirmation is the §8 contract. The UI (PR-037) will gather the serial via a typed input. The backend MUST reject mismatch even when called via curl; defense in depth.

The `_load_latest_drive` helper queries the `pd_snapshots` table for the most recent row matching `enclosure_id=e, slot_id=s`. If multiple rows exist with the same e:s across snapshots (which is normal; new snapshot every 300 s), pick the latest by `captured_at`. Document the assumption: the latest snapshot is at most 5 minutes stale; the operator's information is at most that old. If real-time accuracy matters, the operator triggers a manual refresh first (separate concern).

The audit event message includes the serial number for forensic traceability. After-the-fact, the audit shows: "operator alex offlined drive 2:0 serial WD-XXXXXX". Without the serial, identifying which physical drive was acted on is impossible if e:s mappings later change.

`asyncio.to_thread(runner.run, argv)` is required because the storcli call is sync (subprocess) and blocking. Without it, the FastAPI event loop pauses for the duration of the call, blocking other requests.

The command builders use f-strings with integers. The integers are validated to be 0..255 before f-string interpolation, so injection is structurally impossible at this layer; the runner whitelist is belt-and-suspenders.

Risk heading in PR description: this PR touches the storcli wrapper, security model, and emits destructive commands. Risk text: "If validation fails open or the wrong drive is selected, RAID 6 array can lose its second redundancy and a subsequent drive failure causes data loss. Mitigation: serial-number confirmation, state machine, hand-verification by operator on production hardware before declaring merge-ready."
