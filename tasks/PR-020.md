## Problem

The current dashboard "hero block" wastes the top fold of a 1080p viewport: a large summary value, broad whitespace, and a separate badge component, all expressing what the §3 design system says should be a six-tile horizontal strip. The roadmap §1 principle is "one screen, no scroll": the operator should see system health within two seconds without scrolling.

This PR replaces the hero block with the six tiles enumerated in roadmap §5 / PR-009c: Controller, VD, RAID, BBU, MaxTemp, RoC. Each tile is a `StatusTile` component (PR-019). The view model adds the per-tile derivations; templates compose them.

## Scope

In scope:
- A new view-model section `OverviewStripSection` aggregating six tiles, each with `label`, `value`, `status`, `icon`, `href`.
- Six derivation helpers in `services/overview.py`: controller health, VD aggregate state, RAID aggregate state, BBU presence-and-state, max physical-drive temperature, RoC temperature (already exists from PR-009; reuse).
- Replacement of the existing hero block in `templates/partials/overview_data.html` with a `<div class="status-strip">` containing six `{% include "partials/status_tile.html" with ... %}` calls.
- Each tile links to a relevant detail view: Controller / VD / RAID → `/` (no separate page yet, click is informational), BBU → `/drives` filtered to the BBU-relevant drive (or `/` if not applicable), MaxTemp → `/drives` sorted by temperature desc, RoC → `/` (anchor to RoC tooltip in PR-027).
- View-model tests for each derivation rule.
- Integration test that the rendered overview includes exactly six tiles.

Out of scope:
- Tooltip with current vs threshold values. PR-027.
- Removing the drive table from the dashboard. PR-023.
- Compact alert status row. PR-021.
- Activity timeline. PR-022.

## Implementation notes

The strip is the answer to "one screen, no scroll". On 1080p, six tiles at 180 px min-width plus 8 px gaps fit comfortably under 1320 px (the `--content-max-width` from PR-018). On narrower viewports, `flex-wrap: wrap` lets the strip break into two rows of three; that is the §3 implicit "responsive within reason" clause until PR-063 ships proper mobile layout.

The aggregation rule "any critical wins" is the §3.6 "color with meaning" rule: a degraded VD elevates the RAID tile to warning even if controller is optimal. This is the §1 "two seconds to know if intervention needed" rule.

The BBU tile's neutral state when absent (`bbu_present=False`) is intentional. A homelab without a BBU is not a fault; it is a deployment without that hardware. Showing `None` in neutral keeps the tile honest without raising false alarm.

The MaxTemp tile shows the maximum across drives in the latest snapshot. Why max not average: heat damage is local to the hottest drive; an average masks the hot one with the cool ones. The operator cares about the worst case.

The RoC tile reuses `_load_roc_temperature` from PR-009 without re-implementing thresholds. PR-009 ships the section dataclass; PR-020 maps it into the strip-tile shape. This decoupling means PR-027 (tooltip with thresholds) only touches the strip-tile mapping, not the underlying section.

The `aria-label="System status overview"` on the strip is the screen-reader entry point. Each tile inherits the label/value as accessible text via the partial's default markup.

## Success criteria

1. `python -m ruff check .`, `ruff format --check .`, `mypy src`, `pytest` all pass.
2. Overview page renders exactly 6 elements with class `status-tile`.
3. Each of the 6 tiles has a non-empty label and value.

6. With `bbu_present=False`, the bbu tile is `--neutral` and value is `None`.
7. With drive temps `[42, 51, 58]` and `temp_warning=55`, the max-temp tile is `--warning` with value containing `58`.
8. With no physical drives in snapshot, max-temp tile is `--neutral` with value `Unknown`.
9. The strip uses `flex-wrap: wrap` so it does not horizontal-scroll on narrow viewports.
10. `bash scripts/ci.sh` exits 0.
11. No file outside the explicit lists is modified.
