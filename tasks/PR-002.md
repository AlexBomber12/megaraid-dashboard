# PR-002: Events.notified_at column and notification DAO methods

Branch: pr-002-events-notified-at
- Type: feature
- Complexity: low
- Depends on: PR-001
- Priority: 1
- Coder: any

## Problem

PR-001 delivered the SMTP transport plus the CLI to verify deliverability. The next building block for automated alerting is **state on the events themselves** — the notifier service in PR-003 must be able to ask "which critical events have not yet been emailed?" and "mark this event as notified at this timestamp" without scanning the full table or maintaining a separate ledger.

The `events` table currently has no field representing notification state. Without it, the notifier would need a sidecar table or an in-memory set that does not survive process restart, which is a worse design — events are the natural source of truth and already carry the relevant metadata (severity, category, subject, summary, occurred_at).

This is the **second of four** PRs in the email-alerts series:
1. PR-001: SMTP transport, config, CLI. (Merged.)
2. PR-002 (this PR): `notified_at` column on `events` plus DAO query and update methods.
3. PR-003: Notifier service, APScheduler job, dedup, soft throttle.
4. PR-004: UI status panel surfacing alert state in the overview.

## Scope

This PR adds one nullable column `notified_at` to the `events` table via Alembic migration, plus three DAO methods to query and update it. The column has no default and starts as `NULL` for all existing and future rows; PR-003 is the only writer.

In scope:
- Alembic migration adding `notified_at` column (UTC datetime, nullable, default `NULL`, no index in this PR).
- One field on the `Event` SQLAlchemy model matching the column.
- Three DAO functions:
  - `iter_pending_events(session, *, severity, since)` — generator over events whose `severity` matches and `notified_at IS NULL` and `occurred_at >= since`, ordered ASC by `(occurred_at, id)` so the notifier processes oldest first.
  - `mark_event_notified(session, event_id, sent_at)` — set `notified_at = sent_at` for the given event id, raise if not found, return None.
  - `count_events_notified_since(session, *, since)` — return integer count of events where `notified_at >= since`. Used by PR-003 for soft-throttle accounting.
- Tests at the DAO level using the existing in-memory SQLite fixture pattern.

Out of scope:
- Notifier service logic itself (PR-003).
- Scheduler integration (PR-003).
- UI surface (PR-004).
- Index on `notified_at`. The events table is small (insertions only on rule trigger, retention is indefinite but volume is bounded by detector rules — handful per week realistic). If query latency becomes a problem at scale, an index can be added in a follow-up MICRO PR.
- Backfill of `notified_at` for existing events. They stay `NULL`, which means PR-003 would re-attempt to notify them on first run if they match the threshold and `since` window. PR-003 is responsible for choosing a `since` cutoff that excludes pre-deployment events (typically `process_start_time` or a config knob).
- Severity multi-value queries (`severity IN (...)`). Single severity per call is enough; PR-003 calls once per severity tier if needed.

## Implementation notes

`UTCDateTime` is a project-local type decorator. Check `db/models.py` and existing migrations for the exact import path and constructor signature. Do not invent a new type; reuse what is already there.

The Alembic revision id must chain off the most recent existing revision. To find it, list `migrations/versions/` and read the `down_revision` of each file; the head is the one that no other file references. The new migration's `down_revision` equals the current head's `revision`.

The `Iterator[Event]` return type for `iter_pending_events` is intentional: callers can `break` early when the throttle ceiling is reached without forcing the full query result into memory. mypy strict will accept `Iterator` from `typing` or `collections.abc`; match whatever the existing DAO code uses.

The `sent_at.astimezone(UTC)` conversion before assignment is defense-in-depth: if a caller passes `datetime.now(timezone.utc)` it is already UTC; if they pass `datetime.now(some_other_tz)` the column still ends up storing UTC because `UTCDateTime` enforces it on the way in. The explicit conversion is still good practice — it documents intent.

`Event.occurred_at` and `Event.notified_at` comparisons in SQLAlchemy work correctly because `UTCDateTime` normalizes on bind. The `since` parameter is bound as a TZ-aware UTC datetime.

The DAO does not commit. This keeps `mark_event_notified` composable with the notifier's outer transaction in PR-003 (one commit at the end of the cycle, not per event).

`session.execute(select(...).where(...))` returns a `Result`; chain `.scalars()` to get model instances. For the count query, chain `.scalar_one()` after `select(func.count()).select_from(Event)`.

The migration must be runnable forward and backward. Test by running `alembic upgrade head` and then `alembic downgrade -1` against an empty SQLite DB locally before pushing; CI will catch failures but local verification saves a round trip.
