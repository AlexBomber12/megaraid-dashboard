## Problem

PR-036 ships the destructive replace endpoints. Operators need a guided UI: a multi-step wizard that walks through the procedure, requires explicit confirmation of the drive serial, exposes a dry-run toggle, and makes "cancel" the default focus on every confirmation screen (per §3.1 forgiveness rule).

The wizard ships in two stages: PR-037 (Step 1: offline + missing) and PR-038 (Steps 2 and 3 backend; PR-038's UI extension is bundled there because Steps 2-3 of the wizard live next to Step 1 in the same template). For now, PR-037 covers the Step 1 wizard surface.

## Scope

In scope:
- New section on `templates/pages/drive_detail.html` titled "Replace drive" with a wizard component.
- Step 1 of the wizard:
  1. Confirmation screen showing drive metadata: e:s, model, serial, current state. Two buttons: Cancel (default focus), Continue.
  2. Serial-number input screen: operator types the serial; Continue is disabled until the typed value matches the drive serial (client-side check) AND the dry-run / live toggle is set.
  3. Action screen: invokes `POST /drives/<e>:<s>/replace/offline` then on success `POST .../missing`. Shows step-by-step status.
- Dry-run toggle: a checkbox above the action button. When checked, the body includes `dry_run: true`.
- Result display: success or error, with the storcli response visible for debugging.
- The wizard collapses back to "Replace drive" button after success or cancel.
- Tests: rendered markup, htmx attributes, JS form validation.

Out of scope:
- Step 2 / Step 3 wizard extension. PR-038.
- Step 4 rebuild progress polling. PR-039.
- Reusable wizard component for other flows (locate has its own simpler UI).
- Mobile layout for the wizard. PR-063.

`src/megaraid_dashboard/static/js/replace-wizard.js` (new):
- Around 100 lines.
- State machine: idle → confirm → serial → running → result → idle.
- Serial input enables `Run Step 1` only when `value === expectedSerial`. Both `string.trim()` for tolerance.
- On `Run Step 1`, fetch the offline endpoint with body `{serial_number, dry_run}`. On success, fetch the missing endpoint with the same body. Append results to the output pane between calls.
- Use `fetch()` directly, not htmx, because the wizard needs sequential calls with conditional flow on response status. Each fetch sets the CSRF header from the cookie.

```js
(function () {
  function getCookie(name) {
    const m = document.cookie.match(new RegExp("(?:^|; )" + name.replace(/[$()*+./?[\\\]^{|}-]/g, "\\$&") + "=([^;]*)"));
    return m ? decodeURIComponent(m[1]) : null;
  }

  async function postJson(url, body) {
    const res = await fetch(url, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "X-CSRF-Token": getCookie("__Host-csrf") || "",
      },
      body: JSON.stringify(body),
    });
    const text = await res.text();
    return { ok: res.ok, status: res.status, body: text };
  }

  function init(root) {
    const enclosure = root.dataset.enclosure;
    const slot = root.dataset.slot;
    const expectedSerial = root.dataset.serial;
    const stages = {
      confirm: root.querySelector('[data-stage="confirm"]'),
      serial:  root.querySelector('[data-stage="serial"]'),
      result:  root.querySelector('[data-stage="result"]'),
    };
    const serialInput = root.querySelector('[data-replace-input="serial"]');
    const dryRunInput = root.querySelector('[data-replace-input="dry-run"]');
    const runButton = root.querySelector('[data-replace-action="run-step1"]');
    const output = root.querySelector('[data-replace-output]');
    const openButton = root.querySelector('[data-replace-action="open"]');

    function show(name) {
      Object.values(stages).forEach(function (s) { if (s) s.hidden = true; });
      if (stages[name]) stages[name].hidden = false;
    }

    function close() { show(null); openButton.hidden = false; output.textContent = ""; serialInput.value = ""; runButton.disabled = true; }

    serialInput && serialInput.addEventListener("input", function () {
      runButton.disabled = serialInput.value.trim() !== expectedSerial.trim();
    });

    root.addEventListener("click", async function (evt) {
      const action = evt.target && evt.target.dataset && evt.target.dataset.replaceAction;
      if (!action) return;
      if (action === "open") { openButton.hidden = true; show("confirm"); }
      if (action === "cancel" || action === "close") { close(); }
      if (action === "continue-to-serial") { show("serial"); serialInput.focus(); }
      if (action === "run-step1") {
        runButton.disabled = true;
        show("result");
        const body = { serial_number: serialInput.value.trim(), dry_run: dryRunInput.checked };
        output.textContent = "Running set offline...\n";
        const r1 = await postJson(`/drives/${enclosure}:${slot}/replace/offline`, body);
        output.textContent += JSON.stringify({ status: r1.status, body: r1.body }, null, 2) + "\n";
        if (!r1.ok) { return; }
        output.textContent += "Running set missing...\n";
        const r2 = await postJson(`/drives/${enclosure}:${slot}/replace/missing`, body);
        output.textContent += JSON.stringify({ status: r2.status, body: r2.body }, null, 2) + "\n";
      }
    });
  }

  document.addEventListener("DOMContentLoaded", function () {
    document.querySelectorAll("[data-replace-wizard]").forEach(init);
  });
})();
```

## Implementation notes

The wizard is a vanilla DOM state machine, not a framework component. The dashboard does not have React; introducing it for one flow is overkill. The 100 lines of JS are auditable.

Cancel as default focus is mandatory per §3.1. `autofocus` on the cancel button on every screen ensures Enter-to-cancel by default. The destructive Continue button must be a deliberate click.

The dry-run checkbox defaults to ON. The operator must explicitly toggle it OFF to invoke storcli. This is the "if you forget the toggle, no harm done" disposition: forgetting and clicking Run produces a dry-run output, not a real command.

Serial-number client-side validation only enables the Run button. The server still rechecks (PR-036) on every request; client validation is purely UI ergonomics.

The output pane shows raw JSON of each storcli call. This is intentional for now: the operator should be able to copy-paste the output into a runbook or escalation. Future polish can pretty-print or hide.

`fetch()` over htmx because the flow is sequential (offline THEN missing) with conditional logic between (only proceed if step 1 returned ok). htmx's chaining via response triggers can express this but the JS is shorter and clearer.
