## Problem

After Step 3 launches rebuild, the operator wants to know how long it will take and whether it succeeded. storcli reports rebuild progress via `/c0/eX/sY show rebuild J`, returning percent complete and estimated time remaining. This PR exposes a polling endpoint and a UI progress bar for the duration.

Rebuild on a 4 TB drive typically takes 6 to 12 hours depending on array load. The polling endpoint refreshes every 30 seconds; the UI shows percent + ETA.

## Scope

In scope:
- New endpoint `GET /drives/{e}:{s}/replace/rebuild-status`. No CSRF (GET is idempotent), auth required.
- Command builder `build_rebuild_status_command(enclosure, slot) -> list[str]` returning `[f"/c0/e{e}/s{s}", "show", "rebuild", "J"]`.
- Storcli runner whitelist extended.
- Response parsing: extract `percent_complete: int`, `state: str` (`In progress`, `Complete`, `Not in progress`), `time_remaining_minutes: int | None`.
- A new wizard stage `rebuild` that polls the endpoint via htmx every 30 seconds, renders a `<progress>` element plus the parsed fields.
- Tests: command builder, response parsing across realistic storcli payloads (in-progress, complete, idle).
- An audit event when rebuild transitions to `Complete` (or 100 %), so the events page shows when the procedure ended without operator action.

Out of scope:
- Server-side detection of rebuild completion (push notification). Polling is sufficient.
- Aborting rebuild. Operator runs storcli directly.
- Multi-drive parallel rebuild (RAID 6 can rebuild two drives simultaneously). Single-drive UI for now.

`src/megaraid_dashboard/web/routes.py`:
- Add the GET route:

```python
@router.get("/drives/{enclosure}:{slot}/replace/rebuild-status")
async def drive_rebuild_status(
    enclosure: int, slot: int, request: Request
) -> JSONResponse:
    argv = build_rebuild_status_command(enclosure, slot)
    runner = request.app.state.storcli_runner
    payload = await asyncio.to_thread(runner.run, argv)
    parsed = parse_rebuild_status(payload)
    return JSONResponse({
        "enclosure": enclosure, "slot": slot,
        "percent_complete": parsed.percent_complete,
        "state": parsed.state,
        "time_remaining_minutes": parsed.time_remaining_minutes,
    })
```

## Implementation notes

The 30-second poll matches the dashboard auto-refresh cadence. Faster polling stresses the controller without value (rebuild progress changes by sub-percent per minute on a 4 TB drive); slower polling delays operator awareness on completion.

The audit-on-completion is once-per-rebuild: a guard that queries "is the most recent operator-action audit for this slot already `rebuild complete`?" before writing. Otherwise every poll after 100% writes another audit row.

The `<progress>` HTML element is the right primitive. Browsers render it natively with appropriate accessibility semantics. No need for a custom CSS bar.

Storcli rebuild output format varies across firmware revisions. Test against fixtures captured from the production controller; if a future firmware changes the JSON shape, the parser fails fast (StorcliParseError) and the operator sees the error instead of a misleading "0%".

The hx-trigger="load, every 30s" means the htmx GET fires on element insertion AND every 30 s. The element is created when the wizard advances to the `rebuild` stage, so the polling lifecycle matches the user's attention.

The `Close wizard` button inside the rebuild stage stops further polling (htmx removes the element from DOM and the polling timer with it). The operator can close the wizard once they trust the rebuild is on track and check back later via drive-detail page reload.
