## Problem

Per roadmap §Security, before declaring the project "done" it requires two audit passes. Pass 1 is internal: code review for common vulnerabilities, config review, dependency scan. Pass 2 (PR-055) is external: curl-based attacks. Both produce written documents and remediation MICRO PRs.

## Scope

In scope:
- `docs/SECURITY-AUDIT-2026-Q2.md` covering:
  1. Threat model summary (link to `docs/THREAT-MODEL.md`, written here).
  2. Codebase review: SQL injection, command injection (storcli wrapper), path traversal, XSS in templates, auth bypass, CSRF.
  3. Configuration audit: `.env` permissions, sudoers narrow rule, systemd hardening flags, sqlite ACLs.
  4. Dependency scan: `pip-audit` output, vulnerable transitive deps.
  5. Findings table: each finding gets severity (info/low/medium/high/critical), affected file/line, and a follow-up MICRO PR id (filed separately).
- `docs/THREAT-MODEL.md` describing assets (DB, env file, raid array), trust boundaries (LAN, host filesystem, basic-auth perimeter), threat actors (LAN attacker, compromised user account, local privilege escalation), mitigations.
- A small `scripts/security-scan.sh` that runs `pip-audit`, ruff with security rules enabled (S* via ruff config), and prints a summary.
- Findings filed as MICRO PRs (separate from this PR) per the daemon convention.

Out of scope:
- Active exploitation. Pass 2 covers that.
- Penetration testing tools setup (Burp, ZAP). Manual review with code reading is sufficient for the homelab.

## Files to create

`docs/THREAT-MODEL.md` (around 250 lines):
- Sections: Assets, Trust Boundaries, Actors, Threats (STRIDE table), Mitigations, Open Risks.

`docs/SECURITY-AUDIT-2026-Q2.md` (around 300 lines):

`scripts/security-scan.sh`:
```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

echo "==> ruff (security rules)"
.venv/bin/ruff check --select S --output-format concise src/ || true

echo "==> pip-audit"
.venv/bin/pip-audit --strict || true

echo "==> bandit (deeper SAST)"
.venv/bin/bandit -r src/ -ll || true

echo "==> file permissions on /etc/megaraid-dashboard/env"
stat -c "%a %U:%G %n" /etc/megaraid-dashboard/env 2>/dev/null || echo "env file not present (dev environment)"

echo "==> done"
```

## Implementation notes

The audit document is the deliverable; it is not a code change. The audit may find issues that prompt MICRO PRs, but those PRs are filed separately so this PR is just the documentation.

Findings template inside the audit doc:

```
### Finding 2026-Q2-001
- Severity: medium
- Component: src/megaraid_dashboard/web/auth.py
- Description: bcrypt hash format not validated at config load time; malformed hash crashes mid-request.
- Recommendation: validate format in Settings model_validator.
- Status: filed as MICRO PR (link); merged YYYY-MM-DD.
```

The threat model uses STRIDE (Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege). For a single-operator homelab, most STRIDE categories collapse to "LAN attacker" or "compromised admin credentials".

`pip-audit` checks installed packages against the PyPI vulnerability database. Run quarterly per §6.

`bandit` is the Python equivalent of ruff's `S` rules but more thorough. Some overlap; both run because they catch slightly different things.

## Success criteria

1. `docs/THREAT-MODEL.md` exists and covers assets, boundaries, actors, threats, mitigations.
2. `docs/SECURITY-AUDIT-2026-Q2.md` documents methodology and findings.
3. `scripts/security-scan.sh` runs to completion and produces output.
4. README references the audit.
5. Each finding has a severity, component, description, recommendation, and either an inline fix (cross-reference) or a MICRO PR link.
6. `bash scripts/ci.sh` exits 0 (the audit's fixes do not break CI).
7. No file outside the explicit lists is modified.
