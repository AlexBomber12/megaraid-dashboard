## Problem

The §3 design system is desktop-first per §2.7 and the explicit "out of scope: mobile-first UI" in `AGENTS.md`. However, an operator on the go who gets an alert may want to glance at the dashboard from a phone. A reasonable mobile fallback (responsive at narrow viewport, no separate mobile codebase) is achievable without forking the design.

## Scope

In scope:
- CSS media queries that adapt the existing layout for viewports under 720 px wide:
  - Status strip becomes a 2-column grid instead of 6-wide.
  - Alert status row becomes a 2-column grid.
  - Drives data table becomes a card list (one card per drive) at narrow widths.
  - Events timeline keeps its layout but reduces font size by one step.
  - Header brand and nav collapse into a hamburger toggle (vanilla CSS-only `<details>` element, no JS).
- Touch-friendly tap targets (minimum 44 x 44 px) on all interactive elements.
- No mobile-specific JavaScript.

Out of scope:
- Native app. Web-only.
- iPad-specific layout. The 720 px breakpoint covers both phones and small tablets; for larger tablets the desktop layout works.
- Push notifications. Email is the alert channel.

## Files to modify

`src/megaraid_dashboard/static/css/app.css`:
- Add a single media-query block at the END of the file (after all desktop styles):

```css
@media (max-width: 720px) {
  .status-strip { grid-template-columns: repeat(2, 1fr); display: grid; }
  .alert-status { grid-template-columns: repeat(2, minmax(120px, 1fr)); }
  .data-table thead { display: none; }
  .data-table tr { display: block; padding: var(--space-tight); border: 1px solid var(--surface-border); border-radius: var(--tile-radius); margin-bottom: var(--space-tight); }
  .data-table td { display: block; }
  .data-table td::before { content: attr(data-label) ": "; font-weight: var(--font-weight-label); color: var(--color-neutral); margin-right: var(--space-tight); }
  .button, .filter-chip { min-height: 44px; min-width: 44px; }
  .site-nav { display: none; }
  .site-nav[open] { display: flex; }
}
```

`src/megaraid_dashboard/templates/layouts/base.html`:
- Wrap the nav in a `<details>` element so it collapses on mobile.

`src/megaraid_dashboard/templates/partials/physical_drive_table.html`:
- Add `data-label` attributes on each `<td>` so the mobile card layout's `::before` pseudo-element renders the column name.

`tests/test_web/test_mobile_layout.py`:
- Test: rendered drives page contains `data-label` attributes on `<td>` elements.
- Test: header nav is wrapped in `<details>`.
- Test: no JS file is required for the mobile layout.

## Implementation notes

The card-list-from-table pattern uses `display: block` to break the table grid plus pseudo-elements to inject column labels. The desktop view stays unchanged.

The `<details>` element is the native HTML disclosure. Tap toggles open/closed without JS. Browser support is universal.

44 x 44 px tap targets are the iOS HIG and Material Design minimum. Set on `.button, .filter-chip` covers the dashboard's interactive surfaces.

The 720 px breakpoint covers most phones in landscape and portrait. Tablets in portrait (around 768 px) get the desktop layout; in landscape they get desktop too, which fits since the strip is 6 tiles at around 180 px each.

## Success criteria

1. At viewport width 360 px, the dashboard renders with: 2-column tile grid, 2-column alert-status, drive table as card list, collapsed nav.
2. At viewport width 1080 px, the layout matches the existing desktop view (no regression).
3. All interactive elements at narrow viewport are at least 44 px in either dimension.
4. No new JS file added.
5. Manual smoke: open in Chrome devtools mobile simulator (iPhone 12, viewport 390 x 844), click through dashboard, drives, events, drive-detail.
6. CI passes.
7. No file outside the explicit lists is modified.
