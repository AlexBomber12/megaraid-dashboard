# PR-015: Rate limiting on auth

Branch: pr-015-auth-throttle
- Type: feature
- Complexity: low
- Depends on: PR-005
- Priority: 3
- Coder: any

## Problem

The basic-auth middleware from PR-005 verifies credentials on every request. bcrypt makes brute force expensive (around 100 ms per attempt), but on a LAN with no rate limit, an attacker can issue parallel requests and pin a CPU core. More importantly, the auth log fills with failed-attempt noise, drowning real signal.

Rate limiting at the app layer complements the nginx-level `limit_req` (PR-013). Two layers running the same threshold (5 per minute per IP) means: nginx catches the obvious storm at the edge; the app catches anything that bypasses nginx (LAN attacker hitting port 8090 directly) and produces the application-level audit trail.

## Scope

In scope:
- `slowapi` dependency added to `pyproject.toml`.
- `slowapi` integrated into the FastAPI app: `Limiter` instance keyed by `get_remote_address`, registered as request-state via FastAPI's middleware/handler hooks.
- Rate limit applied to all endpoints that require authentication: 5 attempts per minute per IP. Concretely: any endpoint protected by `BasicAuthMiddleware` that returned 401 within the previous minute counts toward the bucket.
- 429 response on limit exceeded with `Retry-After: 60` header.
- Whitelist: `/healthz`, `/static/*`, `/favicon.ico`. Same as auth and CSRF whitelists.
- Two new config knobs: `auth_rate_limit_per_minute` (default 5) and `auth_rate_limit_burst` (default 2). Validators on both.
- Unit tests covering: under limit passes; over limit returns 429; window expires correctly; whitelisted path is never rate-limited; the limiter keys on `X-Forwarded-For` last value when nginx is in front.

Out of scope:
- Rate limiting on non-auth endpoints. The dashboard's normal traffic is one operator on one IP; broad rate limiting would harm legitimate use.
- Distributed rate-limit storage (redis). In-memory is fine for a single-process deployment; if multi-process arrives, revisit.
- Per-user rate limiting (vs per-IP). One operator account; per-IP is the right key.
- Lockout-on-N-failures with manual unlock. Operationally awkward for a homelab; rate limit alone is sufficient.

`src/megaraid_dashboard/web/auth.py` (the middleware from PR-005):
- After auth verification fails (401 case), record the failed attempt against the limiter. Equivalent of: read settings, look up the limiter from `scope["app"].state.limiter`, manually increment the bucket via slowapi's API (or an explicit decorator on a sentinel route).
- Recommended approach: instead of poking slowapi from the middleware, expose a tiny `/auth-challenge` internal route decorated with `@limiter.limit("{cfg}/minute")` and route the 401 response through it. Cleaner separation; slowapi was designed for routes, not middleware.
- Alternative: a thin standalone middleware `AuthRateLimitMiddleware` that wraps a `defaultdict(deque)` of timestamps per IP and rejects requests that already exceeded the bucket BEFORE calling auth. This avoids slowapi's coupling to FastAPI routes.
- Pick the alternative (standalone middleware): auth lives in middleware, so the rate limit also belonging in middleware is the cleaner pairing. slowapi remains useful as the dependency in case future per-route rate limiting is wanted.

`src/megaraid_dashboard/web/rate_limit.py` (new):
- Implement `AuthRateLimitMiddleware(ASGIApp)`:
  - Read `Settings.auth_rate_limit_per_minute` and `auth_rate_limit_burst` at construction.
  - Internal state: `dict[str, deque[float]]` mapping client IP to deque of attempt timestamps.
  - On every protected request (i.e. not whitelisted), evict entries older than 60 s, count remaining; if count exceeds limit, return 429 with `Retry-After: 60` and a small JSON body `{"error": "rate_limit_exceeded"}`. Otherwise, call the inner app. After the inner app responds, if the response status is 401, record the timestamp; if 200/2xx/3xx, do not record (only failed attempts count toward the bucket).
  - Locking: use `asyncio.Lock` per-IP or a single global lock; for a single-operator homelab the global lock is fine.
- Whitelist mirrors `web/auth.py`. Factor the whitelist constants into `web/_whitelist.py` so all three middlewares share the same list rather than drifting.

`src/megaraid_dashboard/app.py` (continued):
- After importing the new middleware:

```python
app.add_middleware(AuthRateLimitMiddleware)
```

- Order: `ForwardedPrefixMiddleware`, then `AuthRateLimitMiddleware` (rejects floods cheaply), then `BasicAuthMiddleware` (expensive bcrypt), then `CsrfMiddleware`. Rate limit BEFORE auth so an attacker cannot exhaust CPU on bcrypt computations.

## Implementation notes

The "only failed 401s count" rule matters: a logged-in operator browsing rapidly should never trip the limit. Successful auth means the bucket increments only on actual misses, which is the abuse signal we want to throttle.

Memory bound: the per-IP deque grows up to `auth_rate_limit_per_minute + 1` entries before triggering a 429. Old entries are evicted on every check. A periodic janitor that drops entirely empty buckets prevents IP-map growth from a wide scan; implement as a small idle-time pass triggered by a counter (e.g. every 1000 requests, walk the dict and drop empty deques). Optional polish; not essential for first cut.

Burst handling: `auth_rate_limit_burst=2` allows a 2-request burst above the steady rate. The simplest model: the limit is `per_minute + burst` over any 60-second sliding window. If `per_minute=5, burst=2`, allow 7 attempts in 60 s; reject the 8th. Document this clearly so the validator and the implementation agree.

Why not `slowapi` directly in the middleware: slowapi instruments routes via a decorator and reads route-level Limit declarations through FastAPI's dependency injection. Middleware runs before routing, so the routes are not yet matched. The slowapi machinery is therefore a poor fit for middleware-level limiting. Implementing a small purpose-built middleware is cleaner than fighting the framework. Keep slowapi installed for any future per-route limit needs.

The `X-Forwarded-For` parsing: trust ONLY the last entry, on the assumption that nginx is the only proxy and it appends. If the deploy ever fronts with multiple proxies, revisit. Document this assumption in the code.

The 429 body is intentionally minimal. Do not echo the offending IP or the bucket count; that gives attackers free reconnaissance.

## Success criteria

1. `python -m ruff check .`, `ruff format --check .`, `mypy src`, `pytest` all pass.
2. 5 failed auth attempts in under a minute return 401 each; the 6th (or 8th with burst) returns 429 with `Retry-After: 60`.
3. After the window expires, attempts are accepted again.
4. Successful 200 responses do not increment the bucket.
5. `/healthz`, `/static/*`, `/favicon.ico` are never rate-limited.
6. Rate limit middleware runs BEFORE auth middleware in the stack (verified by inspecting `app.user_middleware` order in a test).
7. `Settings.auth_rate_limit_per_minute` and `auth_rate_limit_burst` defaults and validators all behave as specified.
8. `bash scripts/ci.sh` exits 0.
9. Risk heading in PR description acknowledges security perimeter.
10. No file outside the explicit lists is modified.
