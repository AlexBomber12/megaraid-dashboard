## Problem

Steps 2 and 3 of the replacement procedure complete the swap:
- Step 2 is purely physical (operator pulls the old drive, inserts the replacement). No storcli call.
- Step 3 tells the controller to insert the replacement into the missing slot, kicking off rebuild. The storcli command is `/c0/eX/sY insert dg=N array=M row=K` OR `/c0/dN/vM start replacemissingpd array=M row=K` depending on firmware. The roadmap §7 open question 3 commits to "replacement always at same enclosure:slot"; the simpler `insert` form is the right one.

This PR adds the Step 3 backend route plus a wizard extension covering Steps 2 and 3 in the UI. Step 2 has no API call; the wizard simply shows a "Confirm physical swap done" screen between Step 1 and Step 3.

This PR is HIGH RISK because Step 3 is the command that initiates rebuild. A wrong array/row pair on the right slot, or the right command on the wrong slot, can rebuild over usable data. Hand-verification on real hardware is mandatory before merge.

## Scope

In scope:
- Step 3 endpoint: `POST /drives/{e}:{s}/replace/insert` with body `{serial_number, dg, array, row, dry_run}`.
- Command builder `build_insert_replacement_command(enclosure, slot, dg, array, row) -> list[str]`.
- Storcli runner whitelist extended.
- State machine extension: `insert` allowed only when the prior step in the audit trail is `replace step missing`. The route reads the most recent operator-action audit event for this slot to verify.
- DG / array / row values are derived server-side from the latest VD snapshot at time of Step 1, persisted as part of the Step 1 audit message OR re-derived at Step 3 time. Pick one and document.
- Serial-number confirmation against the replacement drive's serial: the operator types the NEW drive's serial; the server verifies it matches the latest snapshot for that e:s slot AND that it is different from the serial recorded in the Step 1 audit message.
- Wizard UI extension: Stage 4 ("physical swap") confirms operator action; Stage 5 ("insert") gathers DG/array/row (read-only from server) plus new-drive serial input; Stage 6 ("result") shows storcli response and points the operator at the rebuild status (PR-039).
- Tests: command builder, route validation, state machine, dry-run, audit emission.

Out of scope:
- Hot-spare promotion (the alternative to manual replace). Out of scope per §7 open question 3 decision.
- Rebuild progress polling. PR-039.
- Aborting an in-progress rebuild. Operator runs storcli directly if they need to.

```python
def build_insert_replacement_command(
    enclosure: int, slot: int, dg: int, array: int, row: int
) -> list[str]:
    _validate_es(enclosure, slot)
    if not isinstance(dg, int) or dg < 0 or dg > 63:
        raise ValueError("dg must be int in [0, 63]")
    if not isinstance(array, int) or array < 0 or array > 63:
        raise ValueError("array must be int in [0, 63]")
    if not isinstance(row, int) or row < 0 or row > 255:
        raise ValueError("row must be int in [0, 255]")
    return [
        f"/c0/e{enclosure}/s{slot}",
        "insert",
        f"dg={dg}",
        f"array={array}",
        f"row={row}",
        "J",
    ]
```

`src/megaraid_dashboard/web/routes.py`:
- Add the insert endpoint:

```python
class InsertRequest(BaseModel):
    serial_number: str
    dg: int
    array: int
    row: int
    dry_run: bool = False


@router.post("/drives/{enclosure}:{slot}/replace/insert")
async def drive_replace_insert(
    enclosure: int, slot: int, body: InsertRequest, request: Request
) -> JSONResponse:
    with request.app.state.session_factory() as session:
        drive = _load_latest_drive(session, enclosure, slot)
        last_audit = _load_last_replace_audit(session, enclosure, slot)
    if drive is None:
        return JSONResponse({"error": "no snapshot for slot"}, status_code=404)
    if drive.serial_number != body.serial_number:
        return JSONResponse(
            {"error": "serial mismatch (replacement drive)",
             "expected": drive.serial_number, "supplied": body.serial_number},
            status_code=409,
        )
    if last_audit is None or "replace step missing" not in last_audit.message:
        return JSONResponse(
            {"error": "must complete replace step missing before insert",
             "last_audit": last_audit.message if last_audit else None},
            status_code=409,
        )
    old_serial_in_audit = _extract_serial_from_audit(last_audit.message)
    if old_serial_in_audit and old_serial_in_audit == body.serial_number:
        return JSONResponse(
            {"error": "supplied serial matches the OUTGOING drive; "
                      "expected the new replacement drive's serial"},
            status_code=409,
        )
    argv = build_insert_replacement_command(enclosure, slot, body.dg, body.array, body.row)
    if body.dry_run:
        return JSONResponse({"dry_run": True, "argv": argv})
    runner = request.app.state.storcli_runner
    result = await asyncio.to_thread(runner.run, argv)
    with request.app.state.session_factory() as session, session.begin():
        record_operator_action(
            session,
            username=request.scope.get("user_username", "unknown"),
            message=(
                f"replace step insert drive {enclosure}:{slot} "
                f"new serial {body.serial_number} dg={body.dg} array={body.array} row={body.row}"
            ),
        )
    return JSONResponse({"step": "insert", "argv": argv, "result": result})
```

- Helper `_load_last_replace_audit(session, e, s) -> Event | None`: query latest operator-action event whose message references `f"drive {e}:{s}"`.
- Helper `_extract_serial_from_audit(message: str) -> str | None`: parse out the `serial XXX` token.

## Implementation notes (READ BEFORE CODING)

**HAND-VERIFICATION MANDATORY.** Every storcli command emitted by this PR must be:
1. Reviewed letter-by-letter against LSI 9270CV-8i firmware 23.34.0-0019 reference.
2. Tested on real hardware in a controlled scenario: a known-failing drive in a known DG/array/row, replaced with a known-good drive of equal or greater capacity. The test must produce a successful rebuild on real hardware.
3. The hand-verification log (commands sent, storcli responses, observed array state before and after) must be pasted into the PR description.
4. Aleksei must explicitly approve the PR after reading the hand-verification log. No daemon-driven merge.

The "supplied serial matches outgoing" check is the safeguard against the most dangerous mistake: typing the OLD serial after physical swap (operator working from notes, did not refresh). Without the check, the route would happily proceed and the controller would insert what it sees (the new drive), but the audit trail would record the wrong serial.

Why server-side derivation of DG/array/row vs operator input: operators do not know these from looking at the drive; they come from the VD topology before the swap. The server reads the latest `vd_snapshots` for the affected DG, finds the array/row pair where the missing slot lives, and returns those values. The operator sees them but does not type them. This eliminates a class of typos.

The audit-message parsing in `_extract_serial_from_audit` is fragile: it depends on the message format from PR-036. Define a stable format constant `f"replace step missing drive {e}:{s} serial {sn}"` and parse with a regex. If the format ever changes, both write and read sides change in lockstep.

The "around 5 seconds" wait for controller detection is empirical from the 9270CV-8i. After hot-insert, storcli `pd list` might briefly show the slot as `Empty` then `UGood`. The wizard's Continue button does not block; the operator visually confirms via storcli or the dashboard's drive-detail refresh before clicking.

The DG / array / row read by the wizard at Step 3 stage entry: simplest is a new GET endpoint `/drives/{e}:{s}/replace/topology` that returns the values. Implement that as part of this PR.

Risk heading in PR description: this PR emits a destructive storcli command that initiates RAID rebuild. Mitigation: serial confirmation, state machine, hand-verification on real hardware before merge.
