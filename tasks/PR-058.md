## Problem

Last of the three deferred docs. INSTALL.md tells the operator how to deploy; RUNBOOK.md tells how to operate; TROUBLESHOOTING.md is a symptom-to-cause table for failures encountered during deploy or runtime. Real failure modes accumulated over the development cycle inform this document; writing it before the system is stable would produce useless content.

## Scope

In scope:
- `TROUBLESHOOTING.md` with sections:
  1. "Service does not start": permission, config validation, port already in use.
  2. "Healthz returns 503": DB unreachable, collector lock contention, scheduler not running.
  3. "No alerts arriving": SMTP credentials, notifier suppress window, alert severity threshold, maintenance mode, throttle saturation.
  4. "Drive shows wrong state": collector cycle delay, snapshot staleness, storcli connectivity.
  5. "Locate LED not blinking": sudoers fragment, storcli wrapper whitelist, controller firmware.
  6. "Replace flow rejects with 409": state machine prerequisite, serial mismatch, prior step missing.
  7. "Migration fails on upgrade": DB lock, missing alembic revision, manual SQL drift.
  8. "Browser shows 401 forever": basic auth caching, password regeneration, browser credential cache.
  9. "Grafana panel shows no data": exporter port firewall, Prometheus scrape failure, label mismatch.
  10. "High RoC temperature without alerts": threshold misconfigured, hysteresis blocking, notifier paused.
- Each section: symptom (what the operator sees), probable causes (ranked by frequency), diagnostic commands, fixes.
- Around 400 lines.

Out of scope:
- Code-level debugging. The ARCHITECTURE doc is the bridge to code reading.
- Vendor-specific issues with the controller hardware. Refer to LSI/Avago/Broadcom docs.

## Files to create

`TROUBLESHOOTING.md`:
- Symptom-cause-fix structure. Operator opens, finds their symptom, follows the cause-and-fix flow.

```
## Service does not start

### Symptom
`systemctl status megaraid-dashboard.service` shows `failed`; recent journal logs show the failure reason.

### Probable causes
1. Config validation error in `/etc/megaraid-dashboard/env`.
2. Port 8090 already bound by another process.
3. Database unreachable or read-only.

### Diagnostics
- `journalctl -u megaraid-dashboard.service -n 50`
- `ss -lnt | grep 8090`
- `sudo -u megaraid sqlite3 /var/lib/megaraid-dashboard/megaraid.db ".tables"`

### Fixes
- Config error: re-run `sudo bash scripts/install.sh --force-reconfigure`.
- Port bound: `sudo lsof -i :8090` and stop the conflicting process.
- DB read-only: `sudo chown megaraid:megaraid /var/lib/megaraid-dashboard/megaraid.db`.
```

## Implementation notes

The 10 sections derive from real-world failure modes encountered during P1 development. A symptom not yet observed should NOT be in this document; speculative entries make the doc harder to navigate without adding value. If a new symptom appears, add a section in a follow-up MICRO PR.

Each diagnostic command should be runnable verbatim; do NOT paraphrase ("check the logs"). The operator under stress wants to copy-paste, not interpret.

Each fix should be the SHORTEST path to recovery. If a deeper investigation is sometimes needed, link to RUNBOOK.md's incident-response section.

## Success criteria

3. INSTALL.md cross-references resolve.

5. Fresh-eyes test: a colleague can recover from at least 3 of the 10 symptoms without external help, using only the doc.
6. CI passes.
