## Problem

Symmetric to PR-065. Consistency check (CC) verifies that parity blocks match data blocks across the array. Discrepancies indicate silent corruption. The controller can run CC automatically on a schedule or on demand; this PR exposes the controls.

## Scope

In scope:
- `GET /controller/consistency-check` returns current state (mode, progress, last run).
- `POST /controller/consistency-check/start` triggers a manual run on a specific VD or all VDs.
- `POST /controller/consistency-check/stop`.
- `POST /controller/consistency-check/mode` body `{mode: "auto" | "manual"}`.
- UI section on the same controller overview as patrol read.
- Audit events.

Out of scope:
- Per-row consistency reporting. The whole-VD pass is what storcli exposes.
- Auto-remediation on inconsistency. The controller default behavior (rebuild parity from data) is the right answer; do not override.

## Files to modify

Same shape as PR-065 (drive_actions, runner, routes, templates, tests).

## Implementation notes

Consistency check runs at lower priority than I/O by default but still consumes bandwidth. The Start endpoint should warn the operator.

Audit messages: `"consistency check start vd N"`, `"consistency check stop"`, `"consistency check mode set to <X>"`.

Emit a `consistency_check_inconsistency` event (severity warning) if the controller reports an inconsistency. The event passes through the regular notifier, so the operator gets emailed if it crosses the alert threshold.
