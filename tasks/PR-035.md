# PR-035: Audit log filter on Events page

## Problem

PR-034 stores operator-action audit events. PR-025 redesigned the events page with filter chips. Combining both: add a filter chip for `operator_action` so the operator (or auditor) can scope the events page to only operator-driven changes. Optionally surface a `/audit` route that pre-filters the events page to category `operator_action` for direct linking from a runbook or bookmark.

## Scope

In scope:
- Add `operator_action` to the filter-chip list on `templates/pages/events.html`.
- A new route `GET /audit` that 302-redirects to `/events?category=operator_action` (preferred over duplicating the page render).
- The `operator_action` chip and the `/audit` route both make `operator_username` visible in the rendered row when present.
- Tests: chip toggles filter; `/audit` redirect lands on the right URL; rendered audit row shows the username.

Out of scope:
- A dedicated audit page with different layout. The events page is sufficient with the filter pre-applied.
- Search by username. The dataset is small (single operator); not worth the UI investment.
- Time-range scoped audit. The events page already supports cursor pagination, which is enough.
- Export.

## Implementation notes

The `/audit` route exists for ergonomics: a bookmark, an operator runbook step, or a Markdown link in `RUNBOOK.md` (PR-057) that says "see /audit for the action history" reads naturally. Without it, the operator has to know the query-param trick.

Why 302 not 301: in case the underlying URL pattern for `events` ever changes; 302 lets us redirect-rewrite without baking a permanent redirect into client caches.

The filter-chip rendering should reuse whatever pattern PR-025 set up. If chips are defined in a Python list of category names, append `operator_action` there; if they are hard-coded in the template, add a row in the template.

The `event-actor` styling is intentionally subdued: italic + neutral color. Operator presence is informational, not alarming. The category badge already signals "this is operator-driven".
