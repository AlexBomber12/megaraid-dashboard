## Problem

PR-032 ships the locate-LED endpoints. Operators still need a UI affordance: a button on the drive-detail page that toggles the locate LED, with a confirmation step (per §3.1 "forgiveness" rule) and a spinner while the request is in flight (per §3.1 "feedback latency budget").

## Scope

In scope:
- A pair of buttons on `templates/pages/drive_detail.html`: "Start locate" and "Stop locate". Visible state shows which is active based on local UI state (not server-polled).
- Confirmation dialog on the start button using a small inline JS confirm or an htmx-driven modal (pick the simpler one for §3.1 "modal dialogs for non-destructive actions: NO"; locate is non-destructive, so an inline `confirm()` or no confirmation is fine; use a small inline confirm for the muscle-memory consistency only).
- Spinner-on-pending: the button gets `aria-busy="true"` and an icon swap (Lucide `refresh-cw` rotating) while the htmx request is in flight.
- Result feedback: on 200, a small inline success indicator next to the button (auto-clears in 5 s); on error, a red status with the error message.
- Tests: rendered drive-detail page contains the buttons; clicking sends POST to the right endpoint; htmx attributes on the button include the CSRF cookie via the shim from PR-014.

Out of scope:
- Real-time LED state. Server does not know the LED state; it only knows the last command sent. Display-only "I sent a start" without polling.
- Multi-drive locate. Single-drive only.
- Touch-friendly large buttons. PR-063 (mobile).

## Files to create

`src/megaraid_dashboard/static/js/drive-actions.js`:
- Around 40 lines.

```js
(function () {
  function confirmIfNeeded(evt) {
    const el = evt.target;
    if (!el || !el.hasAttribute) return;
    const message = el.getAttribute("data-confirm");
    if (message && !window.confirm(message)) {
      evt.preventDefault();
    }
  }

  function flash(el, detail) {
    const fb = el.parentElement.querySelector("[data-action-feedback]");
    if (!fb) return;
    if (detail.successful) {
      fb.textContent = "Sent";
      fb.className = "drive-actions__feedback drive-actions__feedback--ok";
    } else {
      fb.textContent = "Error: " + (detail.xhr && detail.xhr.status);
      fb.className = "drive-actions__feedback drive-actions__feedback--error";
    }
    setTimeout(function () { fb.textContent = ""; fb.className = "drive-actions__feedback"; }, 5000);
  }

  document.body && document.body.addEventListener("htmx:confirm", confirmIfNeeded);

  window.driveActions = { flash: flash };
})();
```

## Implementation notes

`htmx:confirm` is the htmx event for "should this request go through". Returning `evt.preventDefault()` cancels the request. Keeps the confirm flow inline without a separate modal component.

The spinner animation is short (the request is fast, around 200 ms with mocked storcli, around 500-1000 ms with real storcli). The CSS animation uses `var(--transition-fast)` which is 120 ms; the spin appears smooth. If real storcli is slower, raise the duration via a separate token.

The feedback span uses `aria-live="polite"` so screen readers announce success/error without preempting other content. The 5-second auto-clear is per §3.1 "feedback latency budget" and avoids stale messages cluttering the page.

The two buttons are NOT mutually-exclusive in UI state. The operator may click "stop" without ever having clicked "start" (e.g. recovering from a previous session). The endpoints handle either case independently.

The confirmation prompt for "start" is opinionated: locate is non-destructive but the operator should know which slot they are blinking. Better one extra click than mis-pulled drive.

`hx-swap="none"` because we do not replace any DOM; the JS shim handles the feedback. Without `hx-swap="none"` the response body would replace the button.
