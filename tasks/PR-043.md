## Problem

The roadmap §4 commits to P1.5 observability via Prometheus + Grafana. The dashboard is the source of metrics; the homelab-monitoring repo (PR-047 onward) consumes them. This PR ships ONLY the exporter scaffold:
- `prometheus_client` dependency.
- A separate HTTP listener on port 8091 (LAN-only) serving `GET /metrics`.
- A registry plus the bare-minimum metric (`megaraid_exporter_up`) so a Prometheus scrape returns a parseable response from day one.
- Firewall guidance (documentation only; firewall config lives outside the repo).

The actual drive / controller / event metrics land in PR-044, PR-045, PR-046.

## Scope

In scope:
- Add `prometheus-client>=0.20` to dependencies.
- New module `src/megaraid_dashboard/web/metrics.py`. Exposes a `metrics_app` ASGI sub-app that serves the registry.
- Wire `metrics_app` to its own listener on a separate port in `app.py` lifespan: a second uvicorn server in the same event loop, OR a separate Starlette `Mount` exposed on a sub-path. Pick the separate-port approach (port 8091) per roadmap.
- New config field `metrics_port: int = 8091` and `metrics_listen_address: str = "127.0.0.1"`.
- The `/metrics` endpoint is NOT auth-protected (Prometheus expects unauth scrape) but listens only on the LAN interface; document the firewall requirement in README.
- A baseline `megaraid_exporter_up` gauge always set to 1 while the process is up.
- Tests: response body parses as Prometheus text format; `megaraid_exporter_up 1.0` is present.

Out of scope:
- Drive metrics. PR-044.
- Controller metrics. PR-045.
- Event / alert metrics. PR-046.
- Histogram / summary metrics. Counters and gauges only for the homelab.
- Authentication on /metrics. Per Prometheus conventions and the roadmap, the listener is bound to the LAN interface; that is the security boundary.

`src/megaraid_dashboard/web/metrics.py` (new):
```python
from __future__ import annotations

from prometheus_client import CollectorRegistry, Gauge, generate_latest, CONTENT_TYPE_LATEST
from starlette.applications import Starlette
from starlette.requests import Request
from starlette.responses import Response
from starlette.routing import Route


def build_registry() -> CollectorRegistry:
    registry = CollectorRegistry(auto_describe=True)
    up = Gauge(
        "megaraid_exporter_up",
        "1 when the megaraid-dashboard exporter is running.",
        registry=registry,
    )
    up.set(1)
    return registry


async def metrics_endpoint(request: Request) -> Response:
    registry: CollectorRegistry = request.app.state.registry
    body = generate_latest(registry)
    return Response(body, media_type=CONTENT_TYPE_LATEST)


def create_metrics_app() -> Starlette:
    app = Starlette(routes=[Route("/metrics", endpoint=metrics_endpoint)])
    app.state.registry = build_registry()
    return app
```

- Implementation hint:

```python
import uvicorn

if settings.metrics_enabled:
    config = uvicorn.Config(
        create_metrics_app(),
        host=settings.metrics_listen_address,
        port=settings.metrics_port,
        log_level="warning",
        loop="asyncio",
    )
    server = uvicorn.Server(config)
    metrics_task = asyncio.create_task(server.serve())
    app.state.metrics_server = server
    app.state.metrics_task = metrics_task
```

- On shutdown:

```python
if metrics_task is not None:
    server.should_exit = True
    await metrics_task
```

## Implementation notes

The two listeners (web on 8090, metrics on 8091) share an event loop but have independent uvicorn servers. This is the cleanest separation: web traffic and Prometheus scrapes do not interfere.

Binding to 127.0.0.1 by default keeps the metrics endpoint internal. The homelab-monitoring stack (PR-047 onward) runs on the same NAS host; Prometheus scrapes via loopback. If a future deploy moves the monitoring stack to a different host, change `METRICS_LISTEN_ADDRESS` to `0.0.0.0` and add a firewall rule allowing only the monitoring-host IP. The roadmap's "LAN-only firewall rule" is the operator's responsibility outside the repo.

`auto_describe=True` on the registry exposes HELP and TYPE lines for each metric automatically. Prometheus parses those; without them, the scrape still works but the Grafana hover-text is empty.

The `up` gauge sits at 1 unconditionally while the process is alive. Prometheus's own `up` metric (auto-injected by the scrape) reports 0 if the scrape fails for any reason; our `megaraid_exporter_up=1` adds a process-internal alive signal that distinguishes "scrape worked, exporter says it is healthy" from "scrape worked but exporter says it is degraded" (future use).

Why a separate Starlette app rather than a sub-mount in the main FastAPI: the main app has auth, CSRF, ForwardedPrefix middlewares, all of which are wrong for an unauth Prometheus scrape. Separation keeps the surface cleanly different.
