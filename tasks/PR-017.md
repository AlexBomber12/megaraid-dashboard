## Problem

The dashboard logs through `structlog` to stdout, which systemd captures into journald. Without a per-unit cap, journald entries accumulate against the host's global journald config. On a NAS shared with other services (sms-relay, photoprism, megaraid-dashboard, monitoring stack), one chatty unit can dominate disk usage and push out useful log history for the others.

The fix is a drop-in journald override file that pins the unit's journal storage to 200 MB. The file ships in the repo as a sample; PR-031 copies it to the live location during install.

## Scope

In scope:
- A sample drop-in file `deploy/journald-megaraid.conf` with `SystemMaxUse=200M`.
- README sentence pointing at it and showing the install command.
- Mention in the eventual RUNBOOK.md (PR-057) of how to rotate or inspect the unit's journal.

Out of scope:
- File-based logging (separate log file). systemd-journald is the right answer; do not duplicate.
- Custom log levels or structured-log routing. structlog already handles that in-process.
- Per-event log capping. journald handles that natively via the size cap.
- Touching the host-level `/etc/systemd/journald.conf`. The drop-in file is unit-scoped.

## Files to create

`deploy/journald-megaraid.conf`:
```
# Drop-in journald override for megaraid-dashboard.service
# Install path: /etc/systemd/journald@megaraid-dashboard.conf.d/00-retention.conf
# OR per-unit: edit megaraid-dashboard.service to add:
#   [Service]
#   LogNamespace=megaraid-dashboard
# and place this file under /etc/systemd/journald@megaraid-dashboard.conf.d/

[Journal]
SystemMaxUse=200M
SystemKeepFree=500M
RuntimeMaxUse=50M
MaxRetentionSec=30day
```
- Comments explain the placement options. The `LogNamespace` approach is preferred because it isolates the unit's journal entirely; the simpler approach (just adding `SystemMaxUse` to the host journald) caps everything globally and is not what we want.

## Implementation notes

The `LogNamespace` mechanism creates a separate journald instance per namespace, which keeps the per-unit retention from interfering with the host's global journal. Without `LogNamespace`, a `SystemMaxUse` in a drop-in for one unit would still affect the global journald daemon.

`MaxRetentionSec=30day` is the secondary lever: even if the size cap is never hit, anything older than 30 days is rotated out. For a homelab debugging window, 30 days is the right horizon.

The values 200M / 500M / 50M / 30day are tuned for a NAS with hundreds of GB free. The 200M cap holds approximately 2-3 weeks of normal log volume (one collector cycle every 5 minutes plus event detector chatter). If volume is higher in practice, the operator can raise to 500M without consequence.

This PR is operationally important but trivially small. Do not let it grow.
