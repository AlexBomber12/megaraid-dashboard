## Problem

PR-001 delivered the SMTP transport plus CLI, and PR-002 added the `notified_at` column to events plus DAO methods to query and mark them. What is still missing is the **runtime loop** that ties these together: a service that periodically scans for pending critical events, sends emails, marks them notified, and applies dedup plus throttle constraints so the operator inbox does not get flooded during a multi-event incident.

The hardware risk (RoC chip running 107-110 °C while awaiting a permanent heatsink mod) makes this the cornerstone PR of the email-alerts series. After this lands and the production deployment picks it up, the operator gains real out-of-band visibility into critical events without staring at the dashboard.

This is the **third of four** PRs:
1. PR-001: SMTP transport, config, CLI. (Merged.)
2. PR-002: `notified_at` column and DAO methods. (Merged.)
3. PR-003 (this PR): Notifier service plus APScheduler job, dedup, soft throttle.
4. PR-004: UI status panel surfacing alert state in the overview.

## Scope

This PR delivers a single notifier cycle function that the existing APScheduler instance can call every 60 seconds. The cycle:
1. Computes a `since` cutoff from the configured suppression window.
2. Queries pending events at the configured severity threshold (currently `critical`).
3. For each event, checks dedup: if a previous event with the same `(severity, category, subject)` was notified within the suppression window, skip and mark the current event as notified anyway (so it does not pile up).
4. Sends an email via the transport from PR-001.
5. Marks the event notified with the current UTC timestamp.
6. Honors a soft throttle: if more than `alert_throttle_per_hour` emails have been sent in the trailing hour, log a warning and continue sending (per the agreed soft-cap policy: dedup is the primary defense, throttle is observability).

In scope:
- `src/megaraid_dashboard/services/notifier.py` with one public function `run_notifier_cycle(session, transport, *, settings, now)` plus internal helpers.
- A new APScheduler job named `event_notifier` registered alongside the existing collector job, scheduled at a fixed interval (60 seconds).
- File-based lock around the notifier cycle to prevent overlap if a slow SMTP send pushes a cycle past 60 seconds.
- Email body templates (plain text only — HTML deferred to a future PR if needed).
- Tests covering happy path, dedup, throttle warning, partial-failure isolation, and timezone handling.

Out of scope:
- HTML email bodies. The plain-text format used here is sufficient for the operator inbox; HTML is a polish item and can land separately.
- Per-recipient routing (different recipients per severity or category). All emails go to `settings.alert_to` for now.
- Retry-with-backoff on SMTP failure within a cycle. If `transport.send` raises, the event is **not** marked notified, and the next cycle will retry naturally. This is simpler and bounded by the throttle policy.
- Persistent dedup state across process restarts. Dedup is computed fresh each cycle by querying `events` for prior notifications. No separate ledger.
- Notifications for `info` or `warning` severities. Threshold defaults to `critical` and is configurable via PR-001's `alert_severity_threshold`. Lifting it lower is a config-only change after this PR.
- DMARC report ingestion. We do not have an MX for `alert@alexbomber.com` in send-only deployment.

## Files to create

`src/megaraid_dashboard/services/notifier.py`:
- Module-level logger: `_LOG = structlog.get_logger(__name__)`.
- Module-level constant: `_LOCK_PATH_DEFAULT = "/tmp/megaraid-dashboard-notifier.lock"` (override via settings if a project convention exists; otherwise hardcode and document).
- Frozen dataclass `NotifierCycleResult` with fields:
  - `attempted: int` — events the cycle inspected (passed severity filter).
  - `sent: int` — emails actually transmitted.
  - `deduplicated: int` — events skipped due to dedup window match.
  - `failed: int` — events whose `transport.send` raised.
  - `throttle_warning: bool` — True if the trailing-hour count exceeded `alert_throttle_per_hour` during this cycle.
- Public function `run_notifier_cycle(session, transport, *, settings, now) -> NotifierCycleResult`:
  - Validate `now` is timezone-aware UTC; raise `ValueError` if naive.
  - Compute `since = now - timedelta(minutes=settings.alert_suppress_window_minutes)`. This is the cutoff for "pending events older than this are stale and we do not retroactively spam".
  - Query: `pending = list(iter_pending_events(session, severity=settings.alert_severity_threshold, since=since))`.
  - Compute `trailing_hour_start = now - timedelta(hours=1)`.
  - `notified_count = count_events_notified_since(session, since=trailing_hour_start)`.
  - `throttle_warning = notified_count > settings.alert_throttle_per_hour`.
  - If `throttle_warning`, log `notifier_throttle_warning` with `notified_count`, `limit=settings.alert_throttle_per_hour`. Continue processing (soft cap, do not return).
  - For each event in `pending`:
    - Build a dedup key tuple `(event.severity, event.category, event.subject)`.
    - Check dedup: query the events table for any event with the same `(severity, category, subject)` where `notified_at IS NOT NULL AND notified_at >= since`. If found, skip sending, but still call `mark_event_notified(session, event.id, now)` so the event is not re-inspected next cycle. Increment `deduplicated`.
    - Otherwise, build an `AlertMessage`:
      - `subject = f"[MegaRAID {event.severity}] {event.category}: {event.subject}"`. Cap at 200 chars (slice).
      - `body_text` = a multi-line plain-text body with: timestamp (formatted as ISO 8601 UTC plus Europe/Rome via existing `utc_to_cest` helper if importable, else just UTC), severity, category, subject, summary, controller label (`LSI MegaRAID SAS9270CV-8i`), and a trailing line `Do not reply. This mailbox is unmonitored.`.
    - Wrap the send in try/except over `(smtplib.SMTPException, ssl.SSLError)`:
      - On success: `mark_event_notified(session, event.id, now)`, increment `sent`, log `notifier_event_sent` with `event_id`, `severity`, `category`, `subject`.
      - On failure: increment `failed`, log `notifier_event_failed` with `event_id`, `error=str(exc)`, `exc_info=True`. Do NOT mark notified — next cycle retries.
    - Increment `attempted` regardless.
  - `session.commit()` once at the end (not per event).
  - Return `NotifierCycleResult`.
- Helper `_build_alert_message(event) -> AlertMessage` for testability. Pure function, no IO.
- Helper `_event_was_notified_recently(session, event, since) -> bool` for the dedup check. Pure SQL query.

## Implementation notes

The dedup query inside `_event_was_notified_recently` looks like:
```
select(Event).where(
    Event.severity == event.severity,
    Event.category == event.category,
    Event.subject == event.subject,
    Event.notified_at.is_not(None),
    Event.notified_at >= since,
    Event.id != event.id,
).limit(1)
```
The `Event.id != event.id` clause excludes the candidate itself (in case the event was inserted with a populated `notified_at`, which is not currently the case but defensive). The `.limit(1)` is enough — we only need existence.

The cycle does NOT use the `iter_pending_events` Iterator semantics from PR-002 — it materializes the full pending set with `list(...)` because we need the full count for the cycle result regardless of throttle. This is fine because the realistic pending-event volume is single digits per cycle.

The lock pattern: open the lock file in `os.O_RDWR | os.O_CREAT` mode, call `fcntl.flock(fd, fcntl.LOCK_EX | fcntl.LOCK_NB)`, run the cycle, release with `fcntl.flock(fd, fcntl.LOCK_UN)` and close the fd. If the lock acquisition fails (LOCK_NB), log `notifier_overlap_skipped` and return early with `NotifierCycleResult(0, 0, 0, 0, False)`. This matches the collector's existing lock semantics — copy the helper if there is one, do not invent new lock logic.

The transport's `send` method blocks the cycle for up to 30 seconds per event. If 50 events are pending and SMTP is slow, a single cycle can run multiple minutes. The lock protects against overlap; the next cycle will see the still-pending events on its next tick. This is acceptable given realistic event volumes.

`Europe/Rome` formatting: import `utc_to_cest` from `src/megaraid_dashboard/web/templates.py` if it is a plain function (not Jinja-bound). If it is bound to Jinja's environment, replicate the formatting logic locally inside `notifier.py` — do not import Jinja-only symbols into a service module. The locally-replicated version takes a TZ-aware UTC datetime and returns a string formatted with the Europe/Rome timezone. Using stdlib `zoneinfo.ZoneInfo("Europe/Rome")` is the correct approach (Python 3.9+).

The `body_text` format (one suggested layout — adjust to readability):
```
A critical event was recorded on the MegaRAID controller.

Severity:    critical
Category:    pd_state
Subject:     Drive 252:s3 transitioned to FAILED
Summary:     Drive entered FAILED state from ONLINE.
Occurred at: 2026-04-29 22:14:03 UTC (2026-04-30 00:14:03 CEST)
Controller:  LSI MegaRAID SAS9270CV-8i

Do not reply. This mailbox is unmonitored.
```

The notifier handles its own session lifecycle in the scheduler wrapper. Do NOT pass a session from FastAPI request scope — the scheduler runs in a separate executor. Use the same session factory pattern the collector uses.

Aim for one `session.commit()` at the end of the cycle. If you need to checkpoint mid-cycle (e.g. because the transport is genuinely slow and you want a successful send to be durable before the next attempt), commit per event but log a TODO comment and call it out in the PR description so reviewers can weigh the tradeoff.

Subject capping: `event.subject[:200]` is fine; if the slice happens to land mid-word the email is still legible. Do not try to be clever with word boundaries.

Throttle behavior is "soft": when `notified_count > limit`, log warning and continue. The warning is a single log line per cycle, not per event.

ruff strict flags to expect: `S603` (none here, no subprocess), `BLE001` (the broad except in `_notifier_job`'s outer guard is acceptable with `# noqa: BLE001` and a comment explaining "scheduler executor will not reschedule on uncaught exception"), `PLR0913` (too many arguments — keep `run_notifier_cycle` signature simple, four params is fine).

## Success criteria

1. `run_notifier_cycle` queries pending critical events via `iter_pending_events`, sends one email per event, marks each as notified, and commits once.
2. Severity filter is honored: events below `settings.alert_severity_threshold` are not sent.
3. Events older than `now - alert_suppress_window_minutes` are not sent (treated as stale).
4. Dedup: when a prior event with the same `(severity, category, subject)` was notified within the window, the current event is skipped (no email sent) but its `notified_at` is set so it does not re-appear next cycle.
5. Soft throttle: when trailing-hour notification count exceeds `alert_throttle_per_hour`, a `notifier_throttle_warning` log line is emitted, but sends continue.
6. Per-event failure isolation: when `transport.send` raises, the failure is logged with `exc_info=True`, the event's `notified_at` stays `NULL`, and remaining events in the cycle still process.
7. APScheduler job `event_notifier` runs at 60-second intervals, gated on `collector_enabled`.
8. File lock prevents overlap; an attempted concurrent cycle logs `notifier_overlap_skipped` and returns early.
9. `run_notifier_cycle` raises `ValueError` when `now` is timezone-naive.
10. Email subject is `[MegaRAID {severity}] {category}: {subject}` capped at 200 chars.
11. Email body contains severity, category, subject, summary, controller label, occurrence timestamp in both UTC and Europe/Rome, and a `Do not reply` line.
12. `bash scripts/ci.sh` exits 0.
13. No file outside the explicit `Files to create` and `Files to touch` lists is modified.
14. Total net new code is under 700 lines combined (notifier service plus scheduler hook plus tests plus README).
