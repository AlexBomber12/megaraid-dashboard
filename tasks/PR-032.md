# PR-032: Drive locate LED route + command builder

Branch: pr-032-locate-led-backend
- Type: feature
- Complexity: medium
- Depends on: PR-005, PR-014
- Priority: 4
- Coder: any

## Problem

When a drive faults, the operator must know which physical slot to pull. The MegaRAID locate-LED command (`storcli /c0/eX/sY start locate` / `stop locate`) blinks the slot LED so the operator does not pull the wrong drive. Today no UI surface exists for this; the operator must SSH to the NAS and invoke storcli by hand.

This is the first PR of three for the locate flow (PR-032 backend, PR-033 UI, PR-034 audit category). It exposes two POST endpoints with strict integer validation on the e:s pair, builds the storcli command via a whitelist (no string concatenation of operator-supplied values), and runs it through the existing storcli wrapper. CSRF protection (PR-014) is required because the endpoints are state-changing.

## Scope

In scope:
- Two new endpoints: `POST /drives/{enclosure}:{slot}/locate/start` and `POST /drives/{enclosure}:{slot}/locate/stop`. Path parameters validated as integers; non-integer rejects 400 BEFORE reaching the storcli layer.
- Command builder in a new `services/drive_actions.py`: pure function `build_locate_command(enclosure: int, slot: int, action: Literal["start", "stop"]) -> list[str]` returning the argv tokens the storcli runner will invoke. No string concatenation of the action; lookup against a closed dict.
- Whitelist enforcement at the wrapper boundary: the existing `storcli/runner.py` accepts only commands that match a known template; adding "locate" expands the template list with explicit patterns.
- CSRF-protected via the global middleware (PR-014); auth-protected via PR-005.
- Tests for: integer validation, command-builder output exactly matches expected argv, runner accepts the new template, runner rejects an injection attempt (e.g. action=`"start; rm -rf /"`).

Out of scope:
- UI button. PR-033.
- Audit log of the action. PR-034.
- Locate-LED state polling (is the LED currently blinking). The endpoint is fire-and-forget; the storcli reply confirms the command.
- Multi-drive locate. One drive at a time.

## Files to create

`src/megaraid_dashboard/services/drive_actions.py`:
- Module-level constants:

```python
from typing import Literal

LocateAction = Literal["start", "stop"]

_LOCATE_VERB: dict[LocateAction, str] = {
    "start": "start",
    "stop":  "stop",
}


def build_locate_command(enclosure: int, slot: int, action: LocateAction) -> list[str]:
    if not isinstance(enclosure, int) or enclosure < 0 or enclosure > 255:
        raise ValueError("enclosure must be int in [0, 255]")
    if not isinstance(slot, int) or slot < 0 or slot > 255:
        raise ValueError("slot must be int in [0, 255]")
    if action not in _LOCATE_VERB:
        raise ValueError(f"unknown locate action: {action!r}")
    verb = _LOCATE_VERB[action]
    return ["/c0/e" + str(enclosure) + "/s" + str(slot), verb, "locate", "J"]
```

- The function is pure: no I/O, no side effects, easy to unit-test.
- Type signature uses `Literal` so mypy catches misuse at the call site.

## Files to modify

`src/megaraid_dashboard/web/routes.py`:
- Add two routes:

```python
@router.post("/drives/{enclosure}:{slot}/locate/start", name="drive_locate_start")
async def drive_locate_start(enclosure: int, slot: int, request: Request) -> JSONResponse:
    return await _run_locate(enclosure, slot, "start", request)


@router.post("/drives/{enclosure}:{slot}/locate/stop", name="drive_locate_stop")
async def drive_locate_stop(enclosure: int, slot: int, request: Request) -> JSONResponse:
    return await _run_locate(enclosure, slot, "stop", request)


async def _run_locate(enclosure: int, slot: int, action: LocateAction, request: Request) -> JSONResponse:
    try:
        argv = build_locate_command(enclosure, slot, action)
    except ValueError as exc:
        return JSONResponse({"error": str(exc)}, status_code=400)
    runner = request.app.state.storcli_runner
    result = await asyncio.to_thread(runner.run, argv)
    return JSONResponse({"action": action, "enclosure": enclosure, "slot": slot, "result": result})
```

- FastAPI's path-parameter int parsing handles non-integer rejection automatically (returns 422); add a check or rewrite the path schema if you want 400 specifically.

## Implementation notes

The locate command is non-destructive (LED only). It does not affect drive state, RAID state, or data. Even so, it goes through CSRF protection because we want operator actions to follow a single uniform path: any state-changing endpoint requires the token.

Bounds 0..255 on enclosure and slot are loose. The 9270CV-8i with a 24-bay JBOD has slots 0..23 and enclosure id 2 (typical). Anything outside 0..255 is certainly nonsense; tighter bounds add no safety. Let storcli reject in-range-but-physically-absent slot ids itself.

Whitelist enforcement is the security perimeter for storcli. Without it, a route bug or a future endpoint added without thought could pass operator-controlled strings to storcli. The regex whitelist refuses anything that does not match a known template, so even if a route hands `["foo", "bar", ";", "rm"]` to the runner, the runner refuses.

The runner is the boundary; the route is just a user-facing wrapper. Both layers validating separately is intentional.

Path-parameter syntax `{enclosure}:{slot}` requires the colon to be a literal in the URL. FastAPI handles this; some path-encoding edge cases (URL-encoded `%3A`) work too. Document the URL pattern in the routes' docstrings.

Risk heading in PR description: this PR touches the storcli wrapper contract per AGENTS.md security model rule.
