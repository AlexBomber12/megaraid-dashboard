## Problem

When the controller boots and finds drives with previous-array metadata (foreign config), the array is offline until the metadata is imported (preserve existing layout) or cleared (start fresh). Foreign config import is the operation: tell the controller to adopt the foreign config and bring the array online.

This is HIGHER risk than drive replace: an erroneous import can mount a wrong array layout over working drives. A separate task file with hand-verification mandatory is the right disposition. The roadmap §P2 marks this as "even higher risk than drive replace" and suggests splitting if it grows.

## Note

This PR may be split into 2-3 sub-PRs at scheduling time. The single-task-file form here documents the full flow; daemon may dispatch sub-PRs once execution begins.

## Scope

In scope:
- Detect foreign config: extend the parser (or add a new helper) to read `storcli /c0 /fall show all J` and surface foreign-config presence in the controller snapshot.
- A new event category `foreign_config_detected` (severity warning) when present, suppression 24 h.
- Read endpoint: `GET /controller/foreign-config` returns the current foreign config details (DG count, drives in foreign DGs, sizes).
- Import endpoint: `POST /controller/foreign-config/import` (HIGH RISK, claude). Body requires the foreign-config "FCID" or equivalent identifier the operator types. State machine: only allow when foreign config detected; refuse during active rebuild.
- Clear endpoint: `POST /controller/foreign-config/clear` (HIGH RISK, claude). Body requires explicit type-the-string confirmation `CLEAR FOREIGN CONFIG`.
- UI: a dedicated section on the overview page or a new `/controller/foreign-config` page when foreign config exists. Shows the metadata, offers Import and Clear with multi-step confirmation.
- Audit events on every endpoint call.
- Tests including command-builder fuzzing.

Out of scope:
- Auto-import. Always operator-driven.
- Partial import (some DGs but not others). storcli supports it but UX complexity not justified.

## Implementation notes (READ BEFORE CODING)

**HAND-VERIFICATION MANDATORY before merge.** This PR commands an array-state mutation that cannot be reversed by software.

1. Test against a non-production controller OR a thoroughly-mocked storcli runner returning realistic foreign-config payloads.
2. Document the exact storcli commands tested and their responses in the PR description.
3. Aleksei must explicitly approve before merge.
4. The roadmap §P2 hints this PR may be deferred indefinitely if a foreign-config scenario does not arise in production. If no real-world need surfaces by quarterly review, mark WONTDO.

The commands above (`/fall import`, `/fall delete`) are placeholders pending firmware verification; the actual storcli syntax may differ on the 9270CV-8i. DO NOT GUESS; verify against firmware reference.

The Clear path requires typing `CLEAR FOREIGN CONFIG` literally because semantically it is the most destructive operation in the application: the operator is telling the controller to forget metadata that may be the only path back to data on those drives.

Risk heading in PR description: this PR introduces irreversible array-state mutations.
