# PR-023: Drop drive table from Dashboard

Branch: pr-023-drop-drive-table-dashboard
- Type: refactor
- Complexity: low
- Depends on: PR-022
- Priority: 4
- Coder: any

## Problem

The dashboard currently renders a full physical-drive table at the bottom of the overview page. This violates the §2.4 "no duplicated information" rule: the drive table belongs on `/drives`, where it has space to breathe with sortable columns and per-drive detail. The dashboard summarizes; details live elsewhere.

By the time PR-018 / PR-019 / PR-020 / PR-021 / PR-022 ship, the dashboard has the six-tile strip, the alert-status row, and the activity timeline. The drive table at the bottom is now the only large block left, and it conflicts visually with the new dense layout above. Removing it cleans up the dashboard and pushes operators to `/drives` for that view.

## Scope

In scope:
- Remove the drive-table include from `templates/pages/overview.html` (or `partials/overview_data.html`, wherever it is currently rendered).
- Update the view-model so `OverviewViewModel` no longer carries the drive list. The list is loaded only by the `/drives` route; the dashboard load is faster as a result.
- Update tests that asserted drives appeared on the dashboard; they should now assert drives are NOT on the dashboard but ARE on `/drives`.
- Add a small "X drives, see /drives for detail" hint as a single-line summary if the operator misses the table; phrased as a passing reference, not a button.

Out of scope:
- Drives page redesign. PR-024.
- Removing the drive table from anywhere other than the dashboard.
- Performance benchmarking the dashboard load. The drop is intuitively faster; do not over-engineer measurement.

## Implementation notes

The `drive_count` summary is optional. If the activity timeline (PR-022) and the strip's VD tile (PR-020) already convey enough, drop the hint entirely. Less is more for §1 "one screen, no scroll".

If the existing test data fixtures rely on `vm.physical_drives` being populated, the simplest migration is to add `drive_count: int = 0` and let test fixtures pass `0` if they do not care. Avoid making the field optional; explicit zero is clearer than `None`.

The dashboard load time should drop measurably (one query plus zero per-drive renders), but do not benchmark in CI; the simpler `count(*)` query is faster by inspection.

Removing the drive table is irreversible only in the sense that bringing it back would be another PR. If operators complain in the first week, that is what the §9 quarterly roadmap review is for.
