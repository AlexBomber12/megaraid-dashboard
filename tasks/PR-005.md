# PR-005: Basic auth middleware

Branch: pr-005-basic-auth-middleware
- Type: feature
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The dashboard currently runs without any authentication on `0.0.0.0:8090`. On the production NAS this is a LAN-only listener, but the threat model in `AGENTS.md` calls for HTTP basic auth in front of FastAPI. Adding nginx in front (PR-013) is a separate, later change; until then, the application itself must refuse unauthenticated requests. Leaving an open listener increases blast radius if a LAN host is compromised, and it blocks PR-014, PR-015, PR-032, PR-041, all of which assume an authenticated principal exists.

`config.py` already exposes `admin_username: str` and `admin_password_hash: str`. The hash is stored as a bcrypt digest in `/etc/megaraid-dashboard/env`, mode 600. Verification must use `bcrypt.checkpw` over the encoded password bytes, never plain comparison.

## Scope

In scope:
- Starlette ASGI middleware that validates the `Authorization: Basic` header against `admin_username` plus `admin_password_hash` from `Settings`.
- 401 response with `WWW-Authenticate: Basic realm="megaraid-dashboard"` for missing or invalid credentials.
- Constant-time comparison for the username (using `hmac.compare_digest`) plus bcrypt verify for the password.
- Whitelist of paths that bypass auth: `/healthz`, `/static/*`, `/favicon.ico`. The `/healthz` whitelist is wired now even though the route lands in PR-011, so PR-011 has no auth blocker on merge.
- Unit tests covering the four states: no header, malformed header, wrong credentials, correct credentials.
- Integration tests exercising the wired-up middleware against a live `TestClient`.

Out of scope:
- Session cookies, login forms, "remember me", logout endpoint. Basic auth alone is the contract for now.
- Multi-user support. Exactly one administrator account.
- Password rotation tooling. Operator regenerates the bcrypt hash and edits `.env` directly.
- Rate limiting on auth failures. That is PR-015.
- CSRF protection. That is PR-014.
- nginx-level `Authorization` stripping or pre-validation. That is PR-013.
- Audit logging of failed auth attempts. Will be revisited if security audit (PR-054) flags it.

## Files to create

`src/megaraid_dashboard/web/auth.py`:
- New `BasicAuthMiddleware(ASGIApp)` class following the same pattern as `ForwardedPrefixMiddleware` in `web/middleware.py`.
- Module-level constants: `_REALM = "megaraid-dashboard"`, `_WHITELIST_EXACT = frozenset({"/healthz", "/favicon.ico"})`, `_WHITELIST_PREFIX = ("/static/",)`.
- Helper `_is_whitelisted(path: str) -> bool` (pure function, easy to unit test).
- Helper `_verify_credentials(header_value: str | None, settings: Settings) -> bool`.
  - Decode `Basic <b64>` header, split on first colon, hmac-compare username, bcrypt-verify password.
  - Catch `binascii.Error`, `UnicodeDecodeError`, `ValueError` and return False rather than raising.
  - On invalid bcrypt hash format (passlib `ValueError`), log a structlog error (`bcrypt_hash_invalid`) and return False; do not crash the request.
- The middleware reads `Settings` once at construction time from `app.state.settings`, never on every request.

`tests/test_web/test_auth.py`:
- Pure unit tests of `_is_whitelisted` and `_verify_credentials`.
- Async middleware tests using `httpx.AsyncClient` over an `ASGITransport` wrapped around a stub app. The stub app returns 200 with body `b"ok"` so we know the middleware passed the request through.
- Cases: missing header (401, `WWW-Authenticate` present), malformed header (`Basic notbase64`, `Bearer xxx`, raw base64 without `Basic ` prefix), wrong username, wrong password, correct credentials, whitelisted path with no header, whitelisted path with wrong header (still bypasses).

## Implementation notes

bcrypt verification is deliberately slow (around 100 ms on a modest CPU). On every authenticated request that cost is paid in full. For a homelab with one operator this is acceptable, but it does mean the middleware should run AFTER the `/static/*` whitelist check, never before. The whitelist saves bcrypt cost on every CSS, JS, and font request from the browser.

Constant-time username comparison matters less than constant-time password verification (bcrypt already provides that), but `hmac.compare_digest(provided_username.encode(), settings.admin_username.encode())` is one line and removes a timing oracle. Use it.

The `WWW-Authenticate` header on 401 is what triggers the browser's native credentials prompt. Without it the browser shows an opaque error and the operator has to clear cookies to retry. Include it on every 401 response from this middleware.

`_verify_credentials` must NOT log the password. Log only `auth_failure` with the username (truncated to 64 chars) and the failure reason (`malformed_header`, `unknown_user`, `bad_password`, `bcrypt_hash_invalid`). Successful auth produces no log line; otherwise the log fills up with one entry per static asset.

Header-injection guard: a malicious `Authorization` header could contain control characters. Reject anything where `header_value` after `Basic ` does not match `^[A-Za-z0-9+/=]+$` before passing to `base64.b64decode(strict=True)`.

For tests, do not generate fresh bcrypt hashes inside test functions. Hash once at module load (`_TEST_HASH = bcrypt.hashpw(...)`) and reuse. A bcrypt hash takes around 100 ms per call and a test suite with many auth-using cases would otherwise add several seconds.
