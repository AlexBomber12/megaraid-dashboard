**This PR ships in a NEW repository, not megaraid-dashboard.** Per roadmap §P1.5, the observability stack lives in a separate repo `homelab-monitoring` because the same Prometheus + Grafana stack will serve sms-relay and other future products.

Container images for any custom services will follow the standard pattern: `ghcr.io/alexbomber12/<image-name>`.

For the pipeline-orchestrator daemon flow: this PR opens against `homelab-monitoring`'s `main` branch, not `megaraid-dashboard`'s. The daemon must be configured to dispatch this task against the correct repo.

## Problem

The observability stack needs its own repository because:
1. Multiple products (megaraid-dashboard now, sms-relay later) consume it.
2. Coupling Prometheus / Grafana lifecycle to a single product's release cadence is wrong.
3. Stack-level concerns (Caddy reverse proxy, retention tuning, alert routing) belong in one place.

This PR bootstraps the empty repo with: license, README, Docker Compose skeleton, `.gitignore`, CI workflow placeholder. Subsequent PRs (PR-048..PR-053) fill in scrape config, dashboards, and Caddy.

`docker-compose.yml`:
```yaml
services:
  prometheus:
    image: prom/prometheus:v2.55.0
    container_name: homelab-prometheus
    restart: unless-stopped
    user: "65534:65534"  # nobody:nogroup, matches the volume permissions
    command:
      - "--config.file=/etc/prometheus/prometheus.yml"
      - "--storage.tsdb.retention.time=90d"
      - "--storage.tsdb.path=/prometheus"
      - "--web.listen-address=0.0.0.0:9090"
    volumes:
      - "/srv/homelab-monitoring/prometheus/etc:/etc/prometheus:ro"
      - "/srv/homelab-monitoring/prometheus/data:/prometheus"
    ports:
      - "127.0.0.1:9090:9090"  # loopback only by default
    networks:
      - monitoring

  grafana:
    image: grafana/grafana:11.3.0
    container_name: homelab-grafana
    restart: unless-stopped
    user: "472:472"  # grafana:grafana
    environment:
      GF_SECURITY_ADMIN_PASSWORD: "${GRAFANA_ADMIN_PASSWORD:?GRAFANA_ADMIN_PASSWORD is required}"
      GF_USERS_ALLOW_SIGN_UP: "false"
      GF_INSTALL_PLUGINS: ""
    volumes:
      - "/srv/homelab-monitoring/grafana/data:/var/lib/grafana"
      - "/srv/homelab-monitoring/grafana/provisioning:/etc/grafana/provisioning:ro"
    ports:
      - "127.0.0.1:3000:3000"
    depends_on:
      - prometheus
    networks:
      - monitoring

networks:
  monitoring:
    driver: bridge
```

## Implementation notes

The version pins (`prom/prometheus:v2.55.0`, `grafana/grafana:11.3.0`) are chosen to match the latest stable as of roadmap document version (2026-05-02). Update on each container release if security patches accrue; otherwise leave alone for reproducibility.

Loopback-only port mapping by default is the security disposition. PR-050 adds Caddy in front, which terminates TLS and forwards to these loopback listeners. Until then, the operator accesses Grafana via SSH tunnel.

The `GRAFANA_ADMIN_PASSWORD` env var uses Compose's `:?` syntax to refuse to start without a value. Forces operator to fill `.env` instead of running with the default `admin` password.

The host paths `/srv/homelab-monitoring/...` are deliberate: separate from `/var/lib/megaraid-dashboard/`. The two products share a host but not a data dir.

`user: "472:472"` for Grafana matches the upstream container's UID/GID. Without this, bind-mounted `/var/lib/grafana` is owned by root and Grafana cannot write. Document this in the README under "If you see permission errors".

Future custom services in this stack (none yet, but eventually possibly an alertmanager-replacement) would publish to `ghcr.io/alexbomber12/<name>` per the standard convention.

`AGENTS.md` is copied from megaraid-dashboard so the orchestrator can ingest the same workflow rules. The orchestrator-managed sections (Work Modes, Daemon Mode, etc.) get appended automatically when the repo is onboarded.
