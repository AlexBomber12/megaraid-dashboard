# PR-014: CSRF token middleware (cookie-based)

Branch: pr-014-cookie-token
- Type: feature
- Complexity: medium
- Depends on: PR-005
- Priority: 3
- Coder: any

## Problem

Once the dashboard accepts authenticated POST requests (PR-032 locate-LED, PR-036..PR-039 drive replace, PR-041 maintenance mode), it becomes vulnerable to cross-site request forgery: a malicious page on the LAN, opened in the operator's browser, can submit a form to `https://server.alexbomber.com/raid/drives/2:0/locate/start`, and the browser will helpfully attach the cached basic-auth credentials. Every state-changing endpoint in the dashboard must be CSRF-protected before any of those PRs land.

The roadmap (open question 7) settles on **cookie-based** CSRF tokens (the industry standard double-submit pattern), not header-only tokens. The token is set in a cookie with the `__Host-csrf` prefix, mirrored into the request body or a custom header, and validated server-side per state-changing request. htmx integration uses a small JS shim that auto-injects the cookie value into the `X-CSRF-Token` header on every htmx request.

## Scope

In scope:
- A new ASGI middleware `CsrfMiddleware` that:
  - Issues a cryptographically random 32-byte token (base64url-encoded) and stores it in a cookie named `__Host-csrf` on every GET/HEAD/OPTIONS response that does not already carry one.
  - For every POST/PUT/DELETE/PATCH, requires the request to present the cookie value as the `X-CSRF-Token` header. Rejects with 403 otherwise.
  - Constant-time comparison via `hmac.compare_digest`.
- Cookie attributes: `Secure`, `HttpOnly=False` (the JS shim must read it; the token is not a session secret), `SameSite=Strict`, `Path=/`. The `__Host-` prefix mandates: `Secure`, `Path=/`, no `Domain` attribute.
- A small JS shim `static/js/csrf.js` that:
  - Reads the cookie on page load.
  - Configures htmx's `htmx:configRequest` event to set `X-CSRF-Token` on every request.
- Whitelist: `/healthz`, `/static/*`, `/favicon.ico`. Same as PR-005's auth whitelist.
- Tests covering: GET issues cookie; subsequent GET reuses cookie; POST without header gets 403; POST with mismatched header gets 403; POST with matching header passes; htmx shim injects header.

Out of scope:
- Per-form tokens or one-time tokens. The double-submit cookie pattern is sufficient for our threat model and avoids token rotation churn.
- CSRF on GET requests. By spec, GETs are idempotent and not protected; if the app ever introduces side-effecting GETs, that is a separate bug.
- SameSite=Lax fallback for cross-site nav. The dashboard is single-origin via the proxy; Strict is correct.
- A `<form>` token field for non-JS clients. The dashboard is htmx-driven; if a non-JS POST flow is ever needed, surface a hidden input then.

  - `_HEADER_NAME = "X-CSRF-Token"`

- `_generate_token() -> str`: `base64.urlsafe_b64encode(secrets.token_bytes(_TOKEN_BYTES)).rstrip(b"=").decode()`.
- `_extract_cookie(headers: Headers, name: str) -> str | None`: parse the `Cookie` header.
- The middleware mutates outgoing response headers via the standard "wrap send" pattern. On unprotected methods, after the inner app sends, if no `Set-Cookie` for `__Host-csrf` and no incoming cookie of that name, append a `Set-Cookie` with proper attributes.

`src/megaraid_dashboard/static/js/csrf.js`:
```js
(function () {
  function getCookie(name) {
    const match = document.cookie.match(new RegExp("(?:^|; )" + name.replace(/[$()*+./?[\\\]^{|}-]/g, "\\$&") + "=([^;]*)"));
    return match ? decodeURIComponent(match[1]) : null;
  }

  document.body && document.body.addEventListener("htmx:configRequest", function (evt) {
    const token = getCookie("__Host-csrf");
    if (token) {
      evt.detail.headers["X-CSRF-Token"] = token;
    }
  });
})();
```
- Around 15 lines.

## Implementation notes

The `__Host-` cookie prefix enforces three rules at the browser level: cookie must be set with `Secure`, `Path=/`, and no `Domain` attribute. Browsers reject cookies that violate this. The prefix also prevents subdomain attacks: a cookie set on `evil.alexbomber.com` cannot impersonate one on `server.alexbomber.com/raid/` because the prefix bars cross-subdomain cookies.

`HttpOnly` is INTENTIONALLY NOT SET. The double-submit pattern requires JS to read the cookie value and mirror it into the header. `HttpOnly` would prevent that. The token is not a session secret: knowing the token alone does not authenticate; the attacker also needs to ride the basic-auth header, which the browser only attaches to same-origin requests.

`SameSite=Strict` blocks the cookie from cross-site requests entirely, which closes most CSRF vectors before the middleware even sees them. The middleware is belt-and-suspenders for browsers that ignore SameSite (older or quirky configurations) and for non-browser clients that might be tricked into mirroring cookies.

Token length: 32 bytes (256 bits) is comfortably above guess-resistance threshold. URL-safe base64 of 32 bytes is 43 chars without padding.

The middleware should set the cookie LAZILY: only on responses to requests that did not already present the cookie. Otherwise every GET issues a new cookie and the JS shim races with stale values. The "first-time-only" heuristic is: parse incoming `Cookie` header; if `__Host-csrf` already there, do nothing on the response; if not, generate and set.

For the `htmx:configRequest` event: htmx 1.x exposes `evt.detail.headers` as the mutable headers map for the in-flight request. Setting a key here is honored. Reference: `htmx.org/events/#htmx:configRequest`.

The 403 response on CSRF failure should NOT include the expected token in the body or headers. That defeats the purpose. The body can be plain text `csrf token missing or mismatched`.

For the cookie parsing, do NOT use `Headers["cookie"].split(";")` naively; cookies can have whitespace and quotes. Use `http.cookies.SimpleCookie`.
