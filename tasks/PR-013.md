## Problem

The dashboard listens on `0.0.0.0:8090` over plain HTTP. From the LAN this is acceptable for local debugging, but the operator's actual access path is `https://server.alexbomber.com/raid/` through the existing nginx instance on the NAS (not part of this repo). Three concerns are not yet covered:

1. There is no documented nginx config recipe for the path-based proxy with `X-Forwarded-Prefix`, security headers, HSTS, and rate-limit zone for the auth endpoint.
2. No documentation explains the wildcard `*.alexbomber.com` certificate reuse pattern, so a fresh deploy by anyone other than the original author would fail to mount TLS correctly.
3. No reference exists for how the path prefix interacts with the `ForwardedPrefixMiddleware` already in `src/megaraid_dashboard/web/middleware.py`.

This is a documentation-and-samples PR. nginx itself lives on the NAS host outside this repo. No runtime code in this repo invokes nginx; no nginx package is added as a dependency. The PR exists so that a clean deploy from `INSTALL.md` (PR-056) can wire the proxy correctly without trial and error.

## Scope

In scope:
- A new `docs/PROXY-SETUP.md` walking the operator through: cert reuse, server block, location block, security headers, `X-Forwarded-Prefix /raid`, HSTS, `limit_req` zone for `/raid/login` (or whatever endpoint the auth challenge appears on).
- A sample nginx fragment under `deploy/nginx/megaraid.conf.sample`. Fragment, not a full nginx.conf, because the host file already exists and the operator pastes the relevant block in.
- A short reference in the project README pointing at `docs/PROXY-SETUP.md`.
- Verification checklist at the bottom of `PROXY-SETUP.md`: 5 curl commands the operator runs to confirm the proxy is wired correctly.

Out of scope:
- Runtime code in this repo. No nginx package, no auto-generation of nginx config, no nginx health probing.
- Caddy / Traefik / Apache equivalents. nginx only; the production NAS runs nginx.
- Path-rewriting at the app layer. The middleware already handles `X-Forwarded-Prefix`.
- Certificate issuance automation (certbot, acme.sh). The wildcard cert is already issued and reused; the PR documents reuse, not issuance.
- mTLS or client-cert auth.
- Rate-limit values beyond the auth endpoint. PR-015 handles in-app rate limiting; nginx-level rate limit is a defense-in-depth complement, not a replacement.

## Files to create

`docs/PROXY-SETUP.md`:
- Sections, in order:
  1. **Goals**: what the proxy provides (TLS, path namespace, rate-limited auth endpoint, security headers).
  2. **Prerequisites**: nginx installed, wildcard cert at a known path, dashboard reachable on `127.0.0.1:8090`.
  3. **Cert reuse**: the wildcard `*.alexbomber.com` cert lives at `/etc/letsencrypt/live/alexbomber.com/`. Reuse rather than issue per-subdomain.
  4. **Server block**: full `server { ... }` example for `server.alexbomber.com:443`.
  5. **Location block**: `location /raid/ { ... }` with `proxy_pass http://127.0.0.1:8090/`. Explain the trailing-slash significance: with the slash on `proxy_pass`, nginx strips `/raid/` from the path; without the slash, nginx forwards the prefix verbatim. We use the strip pattern, then re-inject via `X-Forwarded-Prefix /raid` so the app generates correct URL paths.
  6. **Security headers**: `Strict-Transport-Security: max-age=63072000; includeSubDomains` (HSTS), `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: DENY`. Note that we do not set `Content-Security-Policy` here; the app may add its own later.
  7. **Rate limit zone**: `limit_req_zone $binary_remote_addr zone=raid_login:10m rate=5r/m;` plus `limit_req zone=raid_login burst=2 nodelay;` inside a nested `location /raid/login` block (or whatever the auth endpoint becomes).
  8. **Healthz**: dedicated `location = /raid/healthz` that proxies through but skips rate limiting and basic-auth requirement at nginx level (the app already whitelists it).
  9. **Verification checklist**: 5 curl commands.
- Length around 250 lines markdown.

`deploy/nginx/megaraid.conf.sample`:
- Copy-paste-ready nginx fragment matching the doc. Heading comment notes it is a sample; placement under `/etc/nginx/conf.d/` or include from the main nginx.conf.
- Skeleton:

```nginx
# megaraid-dashboard reverse proxy fragment
# Place under: /etc/nginx/conf.d/megaraid.conf
# Reload: sudo nginx -t && sudo systemctl reload nginx

limit_req_zone $binary_remote_addr zone=raid_login:10m rate=5r/m;

server {
    listen 443 ssl http2;
    server_name server.alexbomber.com;

    ssl_certificate     /etc/letsencrypt/live/alexbomber.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/alexbomber.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header X-Frame-Options "DENY" always;

    location = /raid/healthz {
        proxy_pass http://127.0.0.1:8090/healthz;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Prefix /raid;
        proxy_set_header X-Forwarded-Proto https;
    }

    location /raid/ {
        proxy_pass http://127.0.0.1:8090/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header X-Forwarded-Prefix /raid;
        proxy_read_timeout 30s;
    }

    # Rate-limit the auth challenge surface.
    # Adjust the location pattern when the actual auth endpoint settles.
    location ~ ^/raid/(login|auth) {
        limit_req zone=raid_login burst=2 nodelay;
        proxy_pass http://127.0.0.1:8090;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Prefix /raid;
        proxy_set_header X-Forwarded-Proto https;
    }
}

server {
    listen 80;
    server_name server.alexbomber.com;
    return 301 https://$host$request_uri;
}
```

## Implementation notes

The path-based proxy at `/raid/` plus the trailing slash on `proxy_pass http://127.0.0.1:8090/` is the correct combination. nginx strips the matched location prefix and forwards the remainder. The app sees `/` as its root path, which keeps `request.url_for(...)` working. The middleware then prepends `/raid` to outgoing URLs because `X-Forwarded-Prefix` is set.

The verification checklist is the contract for "did the operator wire it correctly". It belongs in the doc, not in CI:

1. `curl -I https://server.alexbomber.com/raid/healthz` returns 200 and `Strict-Transport-Security` header present.
2. `curl -I http://server.alexbomber.com/raid/` returns 301 to https.
3. `curl -i https://server.alexbomber.com/raid/` returns 401 with `WWW-Authenticate: Basic realm="megaraid-dashboard"`.
4. `for i in {1..10}; do curl -o /dev/null -s -w "%{http_code}\n" https://server.alexbomber.com/raid/login; done` produces several `429` after the first 5-7 within a minute (rate limit kicked in).
5. `curl -i -u admin:test-password https://server.alexbomber.com/raid/` returns 200 and HTML containing `<a class="brand" href="/raid/">` (the prefix injection worked).

The PR description must include the Risk heading per AGENTS.md because this PR documents the security model perimeter. The risk text: "Misconfigured proxy could expose the app on plain HTTP, leak `Authorization` to logs, or strip security headers. The verification checklist mitigates by being a hard pass-fail before declaring deploy successful."

The `limit_req_zone` rate `5r/m` matches what PR-015 will enforce in-app. Both layers running the same value is intentional defense in depth: nginx rejects bursts at the edge, the app rejects bursts that reach it (which would only happen if a LAN attacker bypasses nginx and hits the app's port directly). Both layers using the same value also makes the operator mental model uniform.

The location pattern for the rate-limit block uses regex (`location ~`) because the actual auth endpoint might be `/raid/login`, `/raid/auth`, or surface only through a 401 challenge on any path. Until PR-005 lands and tests pin down the URL surface, the regex covers the likely candidates. PR-031 may revisit and tighten.
