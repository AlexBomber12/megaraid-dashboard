# PR-016: Disk space monitoring

Branch: pr-016-free-space-watch
- Type: feature
- Complexity: low
- Depends on: PR-005
- Priority: 3
- Coder: any

## Problem

The dashboard's SQLite DB lives at `/var/lib/megaraid-dashboard/megaraid.db`. As snapshots accumulate (one every 300 s, plus event rows, plus retention windows), the DB file grows. SQLite is reasonably compact, but on a small NAS partition or under unexpected event volume, free space can drop to dangerous levels and the DB can corrupt on a write that hits ENOSPC mid-transaction.

There is no monitoring on this. The collector silently fails, alerts stop being recorded, the operator only finds out at the next manual log inspection. This PR adds a periodic free-space probe that emits warning and critical events through the same event pipeline, so the alert email triggers automatically.

## Scope

In scope:
- New service `services/disk_monitor.py` with a function `check_data_partition_free_space(session, *, settings, now) -> list[Event]`.
- Hourly schedule via APScheduler in `services/scheduler.py`.
- Two thresholds in config: `disk_warning_free_mb` (default 500) and `disk_critical_free_mb` (default 100).
- Event category `disk_space`, severities `warning` and `critical`.
- Hysteresis to avoid event spam: do not re-emit same severity within a 6-hour window.
- Tests for the helper, the scheduler wiring, and the config validators.

Out of scope:
- Multi-partition monitoring. Only the partition containing `database_url` is watched.
- Inode exhaustion check. SQLite stores everything in one file; inode pressure is unlikely on the NAS.
- Auto-cleanup of old snapshots when low. The retention service (`db/retention.py`) handles that on a separate schedule; coupling them would be premature.
- A UI tile. PR-020 may add one later if the operator finds it useful; for now the email alert is enough.

## Files to create

`src/megaraid_dashboard/services/disk_monitor.py`:
- Function `_resolve_data_partition(database_url: str) -> Path`. For SQLite URLs, return the directory containing the DB file. For non-SQLite, raise NotImplementedError (forwards-compatible stub).
- Function `_free_space_mb(path: Path) -> int`. Wrap `shutil.disk_usage(path).free // (1024 * 1024)`.
- Function `check_data_partition_free_space(session, *, settings, now) -> list[Event]`:
  - Resolve partition path; if non-SQLite, log a structlog warning and return empty list.
  - Get free MB.
  - Look up most recent `disk_space` event from DB. If one exists with the same severity within `now - 6h`, suppress.
  - Compose event:
    - free below `disk_critical_free_mb` → severity critical, message `"Free space on data partition: {free} MB (below critical threshold {threshold} MB)"`.
    - else if free below `disk_warning_free_mb` → severity warning, similar message.
    - else if a previous warning/critical exists and free is now above warning threshold + 50 MB hysteresis → emit a clear event (severity info, message `"Free space recovered: {free} MB"`).
    - else → return empty.
  - Return list with the single event.

`tests/test_services/test_disk_monitor.py`:
- Mock `shutil.disk_usage` with various values; assert correct severity is returned.
- Test the 6-hour suppression: when a recent same-severity event is in the DB, suppress.
- Test recovery: after a warning event, free space climbs back above threshold + hysteresis, info event emitted.
- Test that a non-SQLite database_url returns empty list and logs.

## Implementation notes

`shutil.disk_usage(path)` works on any path within the partition; it returns the partition's totals, not the directory's. Calling it on the DB file's parent dir is fine.

Why hourly and not 5-minute: a stable system loses MB slowly. Hourly granularity gives a 1-hour worst-case detection delay against a 100 MB critical threshold; even a runaway log loop fills slower than that. 5-minute checks would be needless wakeups.

The 6-hour suppression mirrors the alert-suppression pattern in the notifier. The notifier itself also suppresses; running both is intentional. Disk-monitor's local suppression prevents the events table from filling with one entry per hour for a sustained alert; the notifier's suppression prevents email spam. Two layers, two purposes.

The recovery (info-level clear) event is symmetric with the drive-temperature recovery pattern used in PR-008. Operator visibility on "the disk is fine now" is worth one info-level row.

For non-SQLite database_url, the function returns an empty list rather than raising. The dashboard supports SQLite as the only production target; the non-SQLite path is a forward-compat stub that does not crash if someone experimentally points the app at Postgres.

Resolving the path when the URL is `sqlite:///./megaraid.db` (relative): use `Path(parsed.path.lstrip("/")).resolve().parent`. For `sqlite:////absolute/path.db`, use `Path("/" + parsed.path.lstrip("/")).parent`. Test both forms.
