# PR-008: RoC event detector rule + per-category suppress override

Branch: pr-008-roc-event-detector
- Type: feature
- Complexity: medium
- Depends on: PR-007
- Priority: 2
- Coder: any

## Problem

PR-006 stores the RoC temperature; PR-007 configures the thresholds. Neither produces an event, so an overheating controller still sends no alert. This PR closes that gap.

Two rules are needed together:
1. A new event detector rule `roc_temperature` with hysteresis, modeled on the existing physical-drive `temperature` rule. It emits warning and critical events on threshold crossing and clears them on cool-down past the hysteresis band.
2. A per-category override in the notifier so that `controller_temperature` events use a 24-hour suppression window regardless of the global `alert_suppress_window_minutes` setting. Reason: RoC overheating is a slow-moving fault and an hourly reminder while the operator is investigating adds noise without value; the notifier should send once and stay quiet for the rest of the day.

The existing event categories already include `temperature` for physical drives. RoC events use a separate category `controller_temperature` so the operator can filter, the per-category override can target only RoC, and the event log distinguishes the two clearly.

## Scope

In scope:
- New detector rule `roc_temperature` in `services/event_detector.py`. Reads `roc_temperature_celsius` from the most recent `ControllerSnapshot` and the previous one. Emits events under category `controller_temperature`.
- Hysteresis logic mirroring the existing drive `temperature` rule: enter warning at `>= warning`, enter critical at `>= critical`, exit either state at value `<= threshold - hysteresis`.
- State persistence: a new key in `controller_snapshots`-derived state, OR a reuse of the existing `pd_temp_state` table extended for controllers, OR a new state mechanism. Pick the simplest approach (see Implementation notes below).
- Notifier per-category override mechanism. A small mapping `_PER_CATEGORY_SUPPRESS_MINUTES = {"controller_temperature": 1440}` consulted before falling back to `settings.alert_suppress_window_minutes`.
- Unit tests for: warm-up (snapshot 1 has no previous), threshold cross up, hysteresis hold, threshold cross down, sustained-above-without-cross-down (only one event), config-driven thresholds.
- Notifier tests: `controller_temperature` event respects 1440-minute window; other categories continue to use the global setting.
- Total target: around 250 lines net new across detector, notifier, tests.

Out of scope:
- UI tile rendering. PR-009.
- Migration for any new state column. The detector should reuse existing state machinery if at all possible (see Implementation notes); a new migration is fine if necessary but is the second-best option.
- Configurable per-category suppression. The 1440-minute value is a code constant in this PR; if it later needs to be tunable, surface it in `Settings` then.
- Multi-controller fan-out. Single controller deploy.
- Email content changes for the new category. The existing email template handles `severity + message + category` uniformly; just verify it renders without crashing for the new category.

## Implementation notes

State persistence approach (read carefully before coding): the existing drive temperature detector uses the `pd_temp_state` table (migration 0002). That table is keyed by `(enclosure_id, slot_id)`. RoC has a single state per controller, not per drive. Three options ranked by simplicity:

1. **Compute state from history on each detect call** (simplest). Look at the previous snapshot's `roc_temperature_celsius` and the current value plus the configured thresholds and hysteresis; emit events based purely on the comparison. No persistent state needed: hysteresis is implicit in "did the value cross the threshold band between these two snapshots?". This works because the collector runs every 300 s and we only care about transitions, not absolute state. Pick this option unless tests reveal an edge case it cannot handle.
2. **Add `roc_temp_state` column to `controller_snapshots`** (medium). Each snapshot records its derived state; the detector compares previous-row state to current value.
3. **New `controller_state` table** (most flexible, most plumbing). Skip unless options 1 and 2 are demonstrably insufficient.

The hysteresis math for option 1: if prev value `<` warning threshold and current value `>=` warning, emit `warning` enter event. If prev `<` critical and current `>=` critical, emit `critical` enter (and downgrade any open `warning` state implicitly). If prev `>=` warning and current `<=` warning - hysteresis, emit clear event. The same applies for critical with critical - hysteresis. The "previous snapshot" comes from a `SELECT ... ORDER BY captured_at DESC LIMIT 2` query the detector already does for the drive temperature rule; reuse the pattern.

Why category `controller_temperature` and not `temperature` (which already exists for drives): the operator should be able to filter on Events page by `controller_temperature` versus drive `temperature` independently. Adding the qualifier in the category name is cheaper than adding a sub-category field.

The 1440-minute (24-hour) override is a deliberate choice: silicon thermal events are slow. If the operator does not act within an hour, repeated reminders just train them to ignore. One alert per day is enough to keep it on the operator's todo list without becoming noise. Document this rationale in a code comment near the constant so the next maintainer does not "fix" it back to 60 minutes.

The notifier per-category map is a constant for now. If two more categories need overrides, surface it as `Settings.alert_per_category_suppress_minutes: dict[str, int]` in a follow-up PR. For one entry, hardcoding is correct.

The detector emits messages in English (matches existing convention). Message format: `"RoC temperature {value} C crossed {threshold_name} threshold ({threshold_value} C)"` where threshold_name is `"warning"` or `"critical"`. Clear messages: `"RoC temperature {value} C cleared {threshold_name} threshold"`.

If `roc_temperature_celsius` is `None` in the current snapshot but was `int` in the previous one (e.g. firmware downgrade or sensor failure mid-flight), do NOT emit a clear event. Treat None as "no signal" and skip silently. Emit a structlog `info` line `roc_temperature_unavailable` once per occurrence.
