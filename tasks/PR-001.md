## Problem

The dashboard already records hardware events (`controller`, `cv_state`, `pd_state`, `vd_state`, `smart_alert`, `temperature`) into the `events` table via `record_event` in `db/dao.py`, and the read-only Events page exposes them through HTMX. However, **no out-of-band notification path exists**. If the operator does not open the dashboard, a critical event such as a physical drive entering FAILED state, the cachevault dropping below the capacitance threshold, or a controller temperature crossing the critical bound goes unseen until the next manual check.

The current production controller (LSI MegaRAID SAS 9270CV-8i installed in NAS at `192.168.50.2`) has the RoC chip running 107-110 °C while a temporary heatsink mod is in place, with a permanent heatsink-and-PWM-fan kit on order. Until that hardware fix lands, an unattended overheat event followed by automatic cachevault disable (and consequent write-back cache loss) is a realistic risk. Email alerts are the cheapest mitigation that does not require new hardware.

The configuration scaffolding is partially in place: `Settings` in `src/megaraid_dashboard/config.py` already declares the required fields `alert_smtp_host`, `alert_smtp_port`, `alert_smtp_user`, `alert_smtp_password`, `alert_from`, and `alert_to`. There is no transport implementation, no CLI to verify deliverability, and no notifier-related runtime configuration (TLS mode, severity threshold, suppression window, throttle ceiling).

This PR is the **first of four** in the email-alerts series:
1. PR-001 (this PR): SMTP transport, configuration extensions, and CLI test command.
2. PR-002: `notified_at` column on `events` plus DAO methods to query and mark events.
3. PR-003: Notifier service module plus APScheduler job, dedup, soft throttle.
4. PR-004: UI status panel surfacing last alert send time and pending alert count in the overview view.

## Scope

This PR delivers the transport layer plus a manual verification path. After merge, the operator can run `python -m megaraid_dashboard.alerts test` from the production host and confirm that Proton SMTP credentials, DNS records (SPF, DKIM, DMARC), and outbound network access are all working end-to-end before any automated notifier code lands.

In scope:
- Four new optional fields on `Settings` with defaults and validation.
- A new top-level module `src/megaraid_dashboard/alerts/` containing transport, public exports, and CLI entrypoint.
- A frozen `AlertMessage` dataclass and an `AlertTransport` Protocol so PR-003 can swap in a fake transport for tests without touching the real SMTP path.
- A concrete `SmtpAlertTransport` implementing the Protocol against `smtplib.SMTP` with STARTTLS over port 587 (the only mode currently used by Proton SMTP submission).
- A `python -m megaraid_dashboard.alerts test [--to ADDRESS]` CLI that synchronously sends one fixed test email and surfaces failures with user-friendly messages plus distinct exit codes.
- Tests at the boundary (mock `smtplib.SMTP`, no real network).
- A new section in `README.md` documenting the email-alerts configuration block, the CLI, and an example `.env` aimed at Proton SMTP.

Out of scope:
- The `notified_at` column on `events`, any DAO changes, any migration. Comes in PR-002.
- Any APScheduler job, any wiring of the transport into the live process. Comes in PR-003.
- Any UI surface for alert state. Comes in PR-004.
- SMTPS over port 465 (implicit TLS). Adds a separate code path; not used by Proton submission.
- Dedup, suppression-window enforcement, or throttle counting. Settings are added but **not consumed** in this PR; they are read by the notifier in PR-003.
- Mass-mailer, retry-with-backoff, queue persistence, or fallback transport. YAGNI for a single-controller homelab.

## Files to create

`src/megaraid_dashboard/alerts/__init__.py` — public exports only:
- Re-export `AlertMessage`, `AlertTransport`, `SmtpAlertTransport`.
- Provide a factory `build_default_transport(settings: Settings) -> SmtpAlertTransport` that constructs the concrete transport from the global settings instance.
- Do not import from `__main__.py` here (avoids CLI side effects on import).

`src/megaraid_dashboard/alerts/transport.py` — transport implementation:
- Frozen `dataclass` `AlertMessage` with fields `subject: str`, `body_text: str`, `body_html: str | None = None`.
- `typing.Protocol` `AlertTransport` with one method: `def send(self, message: AlertMessage, *, to: str) -> None`.
- Concrete `SmtpAlertTransport`:
  - Constructor takes a `Settings` instance and stores `host`, `port`, `user`, `password`, `from_addr`, `use_starttls` as instance attributes (do not store the full `Settings` reference; copy what is needed so the transport is decoupled from config reload semantics).
  - `send(message, *, to)` builds an `email.message.EmailMessage`:
    - `Subject` from `message.subject`.
    - `From` formatted as `MegaRAID Alerts <{from_addr}>` using the `email.utils.formataddr` helper (do not concatenate strings, formataddr handles RFC-compliant quoting).
    - `To` from the `to` argument (do not read `settings.alert_to` here — the caller owns recipient selection so PR-003 can route differently per severity).
    - `Date` via `email.utils.formatdate(localtime=True)`.
    - `Message-ID` via `email.utils.make_msgid` with the domain part of `from_addr` (split on `@`, take the right side; if `from_addr` has no `@`, fall back to `make_msgid()` default).
    - Text body via `set_content(message.body_text)`.
    - If `message.body_html is not None`, attach an HTML alternative via `add_alternative(message.body_html, subtype="html")`.
  - Open `smtplib.SMTP(host, port, timeout=30)` as a context manager:
    - `ehlo()` first.
    - If `use_starttls` is True, call `starttls(context=ssl.create_default_context())` then `ehlo()` again.
    - `login(user, password)`.
    - `send_message(msg)`.
  - Wrap the entire flow in a `try` / `except (smtplib.SMTPException, ssl.SSLError) as exc`. On failure log `alert_smtp_failed` with `structlog.get_logger(__name__)` including `subject`, `to`, `error` (str of exc), `exc_info=True`, then re-raise. Do not catch broad `Exception`.
  - On success, log `alert_smtp_send` with `subject`, `to`, `elapsed_ms` (use `time.monotonic()` around the SMTP block), and `starttls` boolean.
  - Use `structlog.get_logger(__name__)` once at module level, not per call.

`src/megaraid_dashboard/alerts/__main__.py` — CLI entrypoint:
- `argparse` with one subcommand `test`.
- `test` accepts optional `--to ADDRESS`. If provided, it overrides the recipient. Otherwise the recipient is `settings.alert_to`.
- The test message:
  - subject: `[megaraid-dashboard] SMTP test`
  - body_text: a multi-line string containing the literal phrase `This is a test message`, the configured `alert_from`, and the local hostname from `socket.gethostname()`. Mention that the message was sent by the CLI test command and is safe to ignore.
  - body_html: `None`.
- Behavior:
  - On success: print `Sent test alert to {to_address}` to stdout, exit 0.
  - On `smtplib.SMTPAuthenticationError`: print `Authentication failed: check ALERT_SMTP_USER and ALERT_SMTP_PASSWORD` to stderr, exit 1.
  - On any other `smtplib.SMTPException`: print `SMTP error: {exception type}: {exception message}` to stderr, exit 1.
  - On `ssl.SSLError`: print `TLS error: {exception message}` to stderr, exit 1.
  - On any other unexpected exception: do not catch, let argparse and Python defaults produce a traceback (the daemon log will capture it).
- Subcommand dispatch via a dict mapping name to handler function. Unknown subcommand or no subcommand: argparse default behavior (print help, exit 2).
- The CLI is allowed to use `print` (this file is the only place in the alerts module where `print` is permitted).

## Implementation notes

`AlertTransport` is a `typing.Protocol`, not an ABC. mypy strict will catch shape mismatches between `SmtpAlertTransport` and the Protocol; no `@runtime_checkable` decorator is needed for this PR (PR-003 may add it if `isinstance` checks become useful).

`build_default_transport` reads the settings from `get_settings()` (module-level call inside the function body, not at import time). This avoids triggering env-var resolution when `from megaraid_dashboard.alerts import AlertMessage` is imported by tests that do not touch settings.

The CLI imports `build_default_transport` lazily inside the `test` handler, not at module top, so `python -m megaraid_dashboard.alerts --help` works without a populated `.env`.

`time.monotonic()` is used for `elapsed_ms` because it is immune to wall-clock jumps. `int((monotonic_end - monotonic_start) * 1000)` for the log field.

`structlog.get_logger(__name__)` returns a bound logger; assign it once at module level as `_LOG = structlog.get_logger(__name__)`. Do not use `print` in `transport.py`.

When mocking `smtplib.SMTP`, remember to mock the context-manager protocol: `mock_smtp.return_value.__enter__.return_value = mock_smtp_instance`. The transport uses `with smtplib.SMTP(...) as smtp:`.

For the `From` display name, use `email.utils.formataddr(("MegaRAID Alerts", from_addr))`. This handles encoding edge cases. Do not f-string the header.

For `Message-ID` domain extraction: `from_addr.rsplit("@", 1)[1]` if `"@" in from_addr` else `None`. `make_msgid(domain=None)` falls back to socket.getfqdn() which is fine.

The CLI prints `Sent test alert to <address>` only on success. On failure, the success message must NOT be printed even if the transport partially completed. Use a return value or exception flow, not a flag.

ruff strict flags to expect: `BLE001` (blind except), `S603` (subprocess), `TRY003` (long exception messages). The transport code as specified avoids all three. If ruff complains about `S324` (insecure hash) in `make_msgid`, it is a stdlib false positive — `# noqa: S324` with a comment explaining the rationale is acceptable.

## Success criteria

1. `Settings` accepts the four new fields with documented defaults; validator rejects bad values with clear `ValueError` messages.
2. `AlertMessage` is a frozen dataclass; mutation attempts raise `FrozenInstanceError`.
3. `AlertTransport` Protocol matches `SmtpAlertTransport` under mypy strict.
4. `SmtpAlertTransport.send` opens an SMTP connection via context manager, calls `ehlo`, optionally `starttls` then `ehlo` again, calls `login`, calls `send_message`, all in that order, with the timeout set to 30 seconds.
5. The constructed `EmailMessage` has correct `From` (with display name), `To` (from argument, not settings), `Subject`, `Date`, and `Message-ID` headers; HTML alternative is attached only when `body_html` is non-None.
6. `python -m megaraid_dashboard.alerts test` exits 0 on success and prints `Sent test alert to {address}` to stdout.
7. `python -m megaraid_dashboard.alerts test --to other@example.com` overrides the recipient.
8. Authentication failure produces a user-friendly stderr message starting with `Authentication failed:` and exit code 1; generic SMTP error or TLS error produces a clear stderr message and exit code 1.
9. No new third-party dependency is introduced; stdlib only.
10. `bash scripts/ci.sh` exits 0 (runs `ruff check`, `ruff format --check`, `mypy src`, `pytest`).
11. `README.md` has an `Alerts (email)` section documenting all ten env vars, the CLI commands, an example `.env` block aimed at Proton SMTP, and a Risk paragraph.
12. No file outside the explicit `Files to create` and `Files to touch` lists is modified.
13. Total net new code is under 600 lines combined (transport plus CLI plus tests plus README); if implementation grows beyond that bound, stop and request scope reduction in the PR description rather than padding.
