# PR-022: Recent activity timeline

- Type: ux
- Complexity: medium
- Depends on: PR-021
- Priority: 4
- Coder: any

## Problem

The dashboard shows tile-summarized state and alert status, but no narrative of "what happened recently". Operators investigating an incident currently have to navigate to the Events page, lose dashboard context, and back-paginate. A small "last 8 events" timeline directly under the alert-status row gives them the recent narrative without losing the at-a-glance dashboard context.

## Scope

In scope:
- A new `Timeline` component (per §3.3): vertical list, severity icon at left, primary text in middle, timestamp at right.
- A new helper `_load_recent_activity(session, *, limit=8)` returning the latest 8 events ordered by `occurred_at` desc.
- A click on an event category navigates to `/events?category=<cat>` (filter chip pre-applied; the events page must accept this query param, which is verified to already work or stubbed).
- The timeline lives in a new partial `templates/partials/activity_timeline.html`.
- Tests for the loader, the partial render, and the click-through link generation.
- A small empty-state when there are zero events.

Out of scope:
- Severity filtering on the dashboard timeline. The dashboard shows everything; filter on the events page.
- Real-time updates. The 30 s auto-refresh of the dashboard is sufficient.
- Pagination on the dashboard timeline. Eight events is the cap; more goes to the events page.
- Relative timestamps ("5 minutes ago"). The dashboard uses absolute local time via `data-local-time` from PR-010. Future polish could add relative; not now.

## Implementation notes

The "8 events" cap is the §1 "one screen, no scroll" constraint working backward: the dashboard above the timeline takes around 200 px (header) + 100 px (status strip) + 80 px (alert-status row); the remaining viewport on 1080p is around 700 px; at 36 px per timeline row, 8 rows fits with margin.

The category link to `/events?category=X` is the "progressive disclosure" rule from §3.1: the dashboard summarizes; clicking drills in. Pre-applying the filter saves the operator a click.

The `severity_icon` mapping is single-source-of-truth here; if any other component (status tile from PR-019) ever needs the same mapping, factor a shared helper into `services/events.py` or `web/templates.py`.

The empty state copy is intentionally short. Operators who see "No events yet" know the system is fresh; they do not need a paragraph explaining what would appear here.

The 30 s overview auto-refresh re-renders this partial; no additional polling needed. htmx triggers `htmx:afterSwap`, which the PR-010 JS shim already uses to re-localize timestamps.
