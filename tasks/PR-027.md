# PR-027: Polish: slot links + tooltip

- Coder: any

## Problem

Two small UX gaps remain after PR-024 / PR-025 / PR-026:

1. Slot ids in the drive table are clickable links, but the same id appearing in the activity timeline (PR-022) or events list (PR-025) is plain text. Operators reading "PD 2:0 degraded" in the timeline cannot click through to drive detail; they have to manually navigate.
2. The RoC tile (and other temperature tiles) shows the value but not the thresholds. To know "is 96 C close to the critical threshold or far from it", the operator has to recall the configured values. A small native-tooltip showing `current / warning / critical` resolves this without adding chrome.

## Scope

In scope:
- A small Jinja filter or macro `slot_link(slot_text, e=None, s=None)` that converts a string like `"2:0"` into `<a href="/drives/2:0">2:0</a>`. Applied in the activity-timeline and events-list message rendering wherever the slot is in a known position.
- An HTML `title="..."` tooltip on the RoC tile, MaxTemp tile, and any per-drive temp cell, showing `Current X / Warning Y / Critical Z` for the relevant thresholds.
- Tests for both the slot-link filter and the rendered tooltip text.

Out of scope:
- Custom rich-tooltip component (popup with multiple values, charts). The native `title` attribute is enough for the homelab and avoids JS overhead.
- Slot-link parsing in arbitrary message bodies. The detector messages have a known `e:s` token position; the filter relies on that.
- Changing thresholds. Just surfacing them.

## Implementation notes

The slot-link filter is a small regex `re.match(r"\d+:\d+", text)`. The risk is XSS via the surrounding string. Two safe paths:

1. The filter receives ALREADY-escaped text (caller `| escape | slot_link`). The filter splits on the regex, leaves both sides as-is, and only the matched token is wrapped in `<a>` (which itself contains only digits and a colon, so safe).
2. The filter escapes the input itself, then wraps. Simpler caller-side: `| slot_link | safe`.

Pick option 2 and explicitly call `markupsafe.escape(...)` inside the filter. Test the malicious-input case.

The tooltip uses HTML's native `title` attribute. It works on hover (mouse) but not touch; that is acceptable for a desktop dashboard. If touch support becomes needed, a CSS-only tooltip layer is a follow-up PR (or PR-063 mobile layout).

The tooltip string format is operator-readable English. Avoid abbreviations that confuse: `C` for Celsius, `Warning` and `Critical` capitalized. Keep the format constant across all temp tiles for predictability.

The strip-tile partial's signature change is optional vs adding a wrapper element in the strip include. Adding `tooltip` as a partial parameter is cleaner; document the change in the partial's docstring comment.
