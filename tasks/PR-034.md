# PR-034: Operator action audit category

Branch: pr-034-operator-audit-category
- Type: feature
- Complexity: low
- Depends on: PR-032
- Priority: 4
- Coder: any

## Problem

Operator actions (locate-LED today, drive replace tomorrow, maintenance-mode toggle later) need an audit trail. The events table is the right home: events already power the dashboard timeline, the events page filter, and email notifications. Adding a new category `operator_action` plus a nullable `operator_username` column on `events` lets every operator-driven mutation be recorded with who-did-what-when, visible alongside system events.

## Scope

In scope:
- New event category `operator_action` (string).
- New nullable column `events.operator_username: str | None`.
- Alembic migration `0005_operator_action_username` with both `upgrade()` and `downgrade()`.
- Helper `record_operator_action(session, *, username, message)` in `services/event_detector.py` (or a new `services/audit.py`).
- Wire the helper into the locate-LED routes (PR-032): on successful command run, write an `operator_action` event with `username=request.user` (the basic-auth principal) and `message="locate start drive 2:0"` etc.
- Tests: migration round-trip, helper writes the row correctly, route emits the event, suppression: operator-action events do NOT trigger email notifications by default (their severity is `info`, below the threshold).

Out of scope:
- A separate audit table. Events table is sufficient and surfaces in the UI for free.
- Per-action permission model. Single-operator deploy; if the auth principal can call the route, the action is permitted.
- Tamper-evidence (HMAC chain). Out of scope for homelab; the §6 quarterly security review re-evaluates.

def record_operator_action(
    session: Session, *, username: str, message: str, occurred_at: datetime | None = None
) -> Event:
    occurred_at = occurred_at or datetime.now(UTC)
    event = Event(
        category="operator_action",
        severity="info",
        message=message,
        occurred_at=occurred_at,
        operator_username=username,
    )
    session.add(event)
    session.flush()
    return event

## Implementation notes

The `operator_username` is nullable because every existing event row was written by the system (no operator). New rows from the locate route populate it; future system events leave it NULL.

Migration 0005 is small (one column add). batch_alter_table for SQLite correctness, identical pattern to PR-006 (RoC column).

The auth principal extraction depends on PR-005 storing the username in `scope`. If not, add it here: in `BasicAuthMiddleware`, after successful verification, set `scope["user_username"] = parsed_username`. Document the cross-PR coordination in this PR's description.

The notifier skip is implicit (severity-based), not category-based, but document the assumption: operator-action events are info-level and never cross the alert threshold. If a future operator action needs to alert (e.g. failed force-rebuild), the helper would emit `severity="warning"` or `"critical"` instead, and the notifier would pick it up automatically. No explicit category exclusion needed.

The events page filter chip for `operator_action` lands in PR-035; the audit data is stored from this PR onward, ready to surface.

Risk heading in PR description: this PR touches the DB schema (events.operator_username) per AGENTS.md security model rule.
