## Problem

The LSI 9270CV-8i ships an on-die temperature sensor for the RAID-on-Chip (RoC). `storcli64 /c0 show all J` exposes it under `Controller -> Response Data -> HwCfg -> ROC temperature(Degree Celsius)` as an integer. This value is the leading indicator for controller throttling and eventual failure: at around 105 C the RoC enters thermal protection and degrades VD performance; sustained operation above 95 C reduces silicon lifetime measurably.

Today the parser does not extract this field, the schema has no column for it, and operators cannot graph or alert on it. PR-007, PR-008, PR-009 all assume this PR has landed; they cover config, alert rule, and UI tile respectively.

## Scope

In scope:
- Extract `ROC temperature(Degree Celsius)` from the storcli `controller show all` payload during parsing.
- Add a nullable integer column `controller_snapshots.roc_temperature_celsius`.
- Alembic migration `0004_controller_roc_temperature.py` with both `upgrade()` and `downgrade()` implemented.
- Wire the parsed value through to `ControllerSnapshot` rows written by the collector.
- Unit tests covering: field present, field absent (legacy storcli), field present with non-integer value (string `"N/A"`).
- Migration tests (or at minimum a manual verification step) that confirm `alembic upgrade head` and `alembic downgrade base` both succeed against a populated SQLite DB.

Out of scope:
- Threshold logic. That is PR-007.
- Event detector rule. That is PR-008.
- UI rendering. That is PR-009.
- Historical backfill of the new column for snapshots taken before the migration. The column stays NULL for those rows; the collector populates it forward only.
- Per-channel temperatures (some controllers expose multiple). The 9270CV-8i exposes one; we read one.

## Implementation notes

The storcli key string `ROC temperature(Degree Celsius)` is verbatim from controller firmware 23.34.0-0019 on the production hardware. Other firmware revisions may emit `ROC temperature(C)`, or may omit the field entirely. Treat any missing or unparsable value as `None` rather than failing parsing; the collector should keep producing snapshot rows even if RoC reading is unavailable.

Pydantic v2 alias matching: storcli keys contain spaces and parentheses. The cleanest approach is a `model_validator(mode="before")` that pulls the RoC key out of the input dict before field-level validation. Field-level `alias=` works but requires `populate_by_name=True` and is fragile when storcli adds variant keys.

The migration file follows the existing naming convention `NNNN_description.py`. Verify the previous migration's revision id by reading `migrations/versions/0003_events_notified_at.py` and copy that exact string into `down_revision`. Do not guess.

`batch_alter_table` is mandatory for SQLite. Without it, Alembic emits a raw `ALTER TABLE ADD COLUMN`, which works for the upgrade but fails on downgrade because SQLite rejects `DROP COLUMN` on tables older than 3.35 (Ubuntu 24.04 ships 3.45 so a direct drop would technically work, but the codebase already uses batch elsewhere for portability and the convention should be kept).

Test isolation for the migration tests: each test creates its own engine via `create_engine("sqlite:///:memory:")` and runs Alembic against it programmatically (`alembic.config.Config` plus `command.upgrade`). Do NOT mutate the production `alembic.ini` path.

The collector currently ignores any field on `ControllerInfo` that does not have a corresponding column. After this PR, `roc_temperature_celsius` is a real column; verify the collector is using `ControllerSnapshot(**parsed_dict)` or an equivalent that picks up the new field, and not a hand-written field list that would silently drop it.
