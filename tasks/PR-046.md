# PR-046: Event + alert metrics

- Type: feature
- Complexity: medium
- Depends on: PR-045

## Problem

PR-044 / PR-045 expose drive and controller state. The remaining picture for the Grafana events dashboard (PR-053) is:
- How many events have been raised, broken down by severity and category (counter, monotonic).
- How many alerts have been emailed (counter).
- How long does the collector cycle take, and when did it last run.

## Scope

In scope:
- Counters in the exporter:
  - `megaraid_events_total{severity, category}` (cumulative count of events raised since process start)
  - `megaraid_alerts_sent_total` (cumulative count of email alerts dispatched)
- Gauges:
  - `megaraid_collector_cycle_duration_seconds` (last cycle duration)
  - `megaraid_collector_last_run_timestamp` (Unix seconds of last successful collector run)
- Counter increment hooks: detector emits a counter increment when it writes an event; notifier emits one when it sends an email; collector wraps each cycle to record duration and last-run timestamp.
- Tests: counters increment correctly; gauges reflect last collector cycle.

Out of scope:
- Per-event-id metrics (high cardinality). Aggregate by severity and category only.
- Histogram of collector duration. Gauge plus structlog log line is enough.
- Notifier latency histogram (time from event emission to email send). Useful but YAGNI for the homelab.

```python
from prometheus_client import Counter, Gauge

EVENTS_TOTAL = Counter(
    "megaraid_events_total",
    "Cumulative count of events raised by the detector.",
    labelnames=["severity", "category"],

    "megaraid_collector_last_run_timestamp",
    "Unix timestamp of the last successful collector cycle.",
)
```

`src/megaraid_dashboard/services/scheduler.py`:
- Wrap the collector job:

```python
async def _run_collector_cycle() -> None:
    start = time.monotonic()
    try:
        await collector.collect_once()
    finally:
        elapsed = time.monotonic() - start
        COLLECTOR_CYCLE_DURATION.set(elapsed)
        COLLECTOR_LAST_RUN_TIMESTAMP.set(time.time())
```

## Implementation notes

Counters never reset within a process. They reset on process restart, which Prometheus handles via `rate()` natively. Operators chart `rate(megaraid_events_total[5m])` instead of the raw counter, so restart blips do not appear as drops.

The maintenance-mode skip in the notifier intentionally does NOT increment `ALERTS_SENT_TOTAL`. Counting "would-have-sent" inflates the metric and makes "did the alert actually go out" hard to derive from Prometheus. A follow-up could add `megaraid_alerts_skipped_for_maintenance_total` if Grafana wants to chart it.

The `time.monotonic()` for duration vs `time.time()` for the last-run timestamp is intentional. Monotonic does not jump on NTP corrections (matters for duration); wall-clock time is what Grafana wants for "when did this run".

The "last run on exception" question: opt for skipping the timestamp update on exception. The metric semantics become "last SUCCESSFUL run", which is what the operator means when they ask "is the collector running?". Errors should be visible separately via structlog and the DB events anyway.

If the collector wraps `collect_once` and a consumer of the existing scheduler job already had retries or its own error handling, do not break that. The wrapper is thin: try/finally around the existing call.
