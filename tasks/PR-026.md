# PR-026: Polish: grammar + badge logic

- Type: bugfix
- Complexity: low
- Depends on: PR-024
- Priority: 4

## Problem

After the §3 redesign lands (PR-018..PR-025), two small but visible polish items remain:

1. The status-strip drive count copy reads "1 drives elevated" when exactly one drive is in a non-optimal state. Singular/plural grammar is wrong.
2. The same drive can simultaneously sit in `warning` and `critical` badges across the strip and the timeline because the derivation rules do not enforce mutual exclusion: a drive at 56 C with `warning` temperature AND `degraded` state aggregates wrong, surfacing two badges for the same condition.

Both bugs are minor in isolation but cumulatively undermine the "two seconds to know if intervention is needed" rule.

## Scope

In scope:
- Singular vs plural copy in `services/overview.py` derivations: any place that emits a count-and-noun phrase uses `_pluralize(n, "drive", "drives")` or equivalent.
- Mutual exclusion rule for badges: critical wins over warning; both win over optimal. Where a drive could match both, render only the higher-severity badge. Document the rule in `_drive_state_badge(...)`.
- Tests for both grammar and exclusion.

Out of scope:
- Localization. Strings stay in English.
- Pluralization for languages other than English.
- Restructuring the overview view model. The polish targets the helper layer, not the dataclass shape.

## Files to modify

`src/megaraid_dashboard/services/overview.py`:
- Add `_pluralize(count: int, singular: str, plural: str) -> str` helper. Use it in every count-emitting helper.
- Refactor any badge-derivation helper to enforce: if both `warning_count > 0` AND `critical_count > 0`, surface only `critical`. Drive table cell logic likewise: `state badge` overrides `temperature badge` when both apply, with the higher severity winning.

`tests/test_services/test_overview_polish.py`:
- Test: `_pluralize(1, "drive", "drives") == "drive"`.
- Test: `_pluralize(0, "drive", "drives") == "drives"` (zero is plural in English).
- Test: `_pluralize(2, "drive", "drives") == "drives"`.
- Test: status-strip aggregation with one drive degraded yields "1 drive degraded" (singular noun, no count word duplication).
- Test: drive-table row at 56 C and `degraded` state emits exactly one badge of severity `critical`-or-`warning` (whichever is higher), not two.
- Test: drive at 56 C and `optimal` state emits one warning badge.
- Test: drive at 50 C and `failed` state emits one critical badge.

## Files NOT to touch

- `services/event_detector.py`. Event derivation has its own logic and is unrelated.
- `services/notifier.py`. Notifier unaffected.
- `db/`, `migrations/`. No schema change.
- Templates. Polish is in the helpers; templates already render whatever the helper returns.

## Implementation notes

The pluralize helper is intentionally simple: 1 yields singular, anything else yields plural. English has irregular cases ("one criterion" / "two criteria") but our nouns are regular: drive, drives; event, events; alert, alerts.

The mutual exclusion rule is a single ordering: critical > warning > info > optimal > neutral. Define this once as a module-level tuple `_SEVERITY_RANK = ("critical", "warning", "info", "optimal", "neutral")` and a helper `_higher_severity(a: str, b: str) -> str` that picks the lower index. All callers route through the helper.

Why this PR is small: both bugs are localized to the helper layer. Templates are unchanged; the copy comes from the helper output. Tests are added but test fixtures from earlier PRs already cover the data shape.

The "0 drives" case in pluralization is contentious in some style guides ("no drives" reads better than "0 drives"). Match the style already used in the dashboard; if the dashboard says "0 events", keep numerals; if it says "no events", switch the helper to "no" for zero. Read existing copy first; do not change copy idiom in a polish PR.
