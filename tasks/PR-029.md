## Problem

PR-028 set up the environment (user, dirs, env file). Stage two creates the Python virtual environment under `/opt/megaraid-dashboard/.venv`, installs the package via `pip install -e .` (or from a built wheel), and runs a basic smoke check (import the package, print version). All idempotent and re-runnable.

## Scope

In scope:
- New `phase_venv` function in `scripts/install.sh`: create `${INSTALL_PREFIX}/.venv` if absent, upgrade pip to a known-good version (24+).
- New `phase_pip` function: clone or copy the source tree into `${INSTALL_PREFIX}/src`; `pip install -e .` against that tree.
- New `phase_smoke` function: as the install user, run `python -c "import megaraid_dashboard; print(megaraid_dashboard.__version__)"` to confirm the package imports.
- Source-tree handling: the script is run from the source checkout (`scripts/install.sh` discovers its own location and uses `..` as source). Copy or rsync the source into `${INSTALL_PREFIX}` excluding `.git`, `.venv`, `__pycache__`.
- pypi reachability check at the start of `phase_pip` (one curl).
- Idempotency: re-run reuses existing venv, re-installs the package (which is fine; pip handles).
- Tests via shellcheck plus a Docker-based smoke (deferred to PR-031 final test if Docker is heavy).

Out of scope:
- Wheel artifact building. PR-059 (versioning + reproducible builds) handles wheels; for now editable install from source is sufficient.
- Configuration wizard. PR-030.
- sudoers / systemd / final smoke. PR-031.
- Network-mirror configuration (corporate pypi proxies). YAGNI.

## Files to modify

`scripts/install.sh`:
- Append phase functions:

```bash
phase_venv() {
  log_info "Phase 4: venv"
  local venv="${INSTALL_PREFIX}/.venv"
  if [[ ! -d "${venv}" ]]; then
    sudo -u "${INSTALL_USER}" python3 -m venv "${venv}"
    log_info "created venv at ${venv}"
  else
    log_info "venv exists, skip"
  fi
  sudo -u "${INSTALL_USER}" "${venv}/bin/pip" install --upgrade pip >/dev/null
}

phase_pip() {
  log_info "Phase 5: pip install"
  local src_dir="${INSTALL_PREFIX}/src"
  local script_dir
  script_dir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
  local repo_root="$(dirname "${script_dir}")"
  if ! curl -fsSI -o /dev/null https://pypi.org/simple/ ; then
    log_fail "pypi.org unreachable; cannot pip install"
  fi
  install -d -m 0750 -o "${INSTALL_USER}" -g "${INSTALL_USER}" "${src_dir}"
  rsync -a --delete \
    --exclude='.git/' --exclude='.venv/' --exclude='__pycache__/' \
    --exclude='*.egg-info/' --exclude='dist/' --exclude='build/' \
    "${repo_root}/" "${src_dir}/"
  chown -R "${INSTALL_USER}:${INSTALL_USER}" "${src_dir}"
  sudo -u "${INSTALL_USER}" "${INSTALL_PREFIX}/.venv/bin/pip" install -e "${src_dir}"
  log_info "package installed"
}

phase_smoke() {
  log_info "Phase 6: smoke"
  local out
  out="$(sudo -u "${INSTALL_USER}" "${INSTALL_PREFIX}/.venv/bin/python" -c \
    "import megaraid_dashboard; print(megaraid_dashboard.__version__)")"
  log_info "imported megaraid_dashboard ${out}"
}
```

## Implementation notes

`pip install -e .` keeps the source tree linked, so `git pull` + restart picks up changes without re-running the installer. For a true production deploy (immutable artifacts), PR-059 will switch to wheel-based install; for the homelab single-host case, editable install from a copied source dir is the simplest workable thing.

The `rsync -a --delete` is destructive within `${INSTALL_PREFIX}/src`. The `--delete` removes files no longer in source. That is intentional: stale `.pyc` files or orphan modules from a prior install are removed. The exclude list keeps `.git` and friends out.

The pypi reachability check uses `curl -fsSI -o /dev/null` to make a HEAD request. If the network is broken, fail fast with a clear message instead of a confusing pip timeout.

Running `pip install` as the install user (via `sudo -u`) keeps the venv ownership clean. Running pip as root would mix root-owned bytecode into a user-owned venv and create permission issues on first run.

The smoke test is just `import megaraid_dashboard`. It catches: missing dependencies, broken setup.py / pyproject build, wrong Python version (already checked in pre-flight, but defense in depth). It does NOT validate the DB or web layer; that is reserved for the final smoke in PR-031 (curl healthz).

## Success criteria

1. `python -m ruff check .`, `ruff format --check .`, `mypy src`, `pytest` all pass.
2. `bash -n scripts/install.sh` parses cleanly.
3. `shellcheck scripts/install.sh` warning-free.
4. Running phases 1-6 on a clean Ubuntu 24.04 with the source tree present completes successfully and prints the version string.
5. Re-running is idempotent: venv reused, source rsync diffs cleanly, pip install -e refreshes.
6. With pypi unreachable (firewall block), phase 5 exits with the documented error.
7. The installed venv at `${INSTALL_PREFIX}/.venv` is owned by `megaraid` user.
8. `bash scripts/ci.sh` exits 0.
9. No file outside the explicit lists is modified.
