# PR-007: RoC config fields + validation

## Problem

PR-006 stores the RoC temperature, but no thresholds exist to evaluate it against. Without configurable warning and critical thresholds, PR-008 cannot emit `roc_temperature` events and PR-009 cannot color the overview tile. The values must be configurable because:
1. Initial deploy uses higher thresholds (95 / 105 C) until the heatsink mod ships.
2. After the 40x40 mm heatsink is installed (manual runbook item), thresholds drop to 80 / 95 C.
3. Different controller revisions in the future may need different defaults.

The existing `Settings` class already has the parallel pair `temp_warning_celsius` / `temp_critical_celsius` for physical drives, with a model validator enforcing critical greater than warning. The RoC pair should follow the same convention.

## Scope

In scope:
- Two new fields on `Settings`: `roc_temp_warning_celsius` (default 95) and `roc_temp_critical_celsius` (default 105).
- A new field `roc_temp_hysteresis_celsius` (default 5) mirroring the existing `temp_hysteresis_celsius` for drives.
- Validator rules: critical greater than warning; both within `[40, 130]`; hysteresis at least 1 and strictly less than warning.
- Unit tests for the validator: each rule violation is caught with a clear `ValueError`.
- Document the new env vars in the project README under the existing config table.

Out of scope:
- Changing the default values for the existing `temp_warning_celsius` / `temp_critical_celsius` fields. Those govern physical drives, not RoC.
- A unified "thermal" section grouping drive and RoC thresholds. Keep the flat structure consistent with the rest of `Settings`.
- Per-controller overrides. Single-controller deploy.
- Loading the new values into the event detector. That is PR-008.

## Implementation notes

The `[40, 130]` range bounds the values to physically plausible silicon temperatures. The lower bound 40 prevents a misconfigured 0 from masking a real hot reading; 130 sits above the 9270CV-8i datasheet shutdown temperature of approximately 125 C. Anything outside that window is almost certainly a typo in `.env`.

The hysteresis rule (must be less than warning) prevents the event detector from oscillating: if hysteresis equaled warning, exiting "warning" state would require dropping below 0 C, which never happens. Mirroring the existing drive temperature rule keeps operator mental model simple.

Pydantic v2 raises `ValidationError` (not `ValueError`) at field-level validation, but our `model_validator(mode="after")` runs custom code that raises `ValueError`. Pydantic wraps that into the same `ValidationError` for the caller. Tests should catch `ValueError` (when constructing `Settings` outside pydantic's normal flow, e.g. via `Settings.model_validate(...)`) using `pytest.raises(ValueError, match=...)` against the message; if the existing tests for the drive thresholds use a different exception class, mirror that exactly.

Defaults are written as plain integers, not strings. `pydantic-settings` reads env vars as strings and coerces; integer defaults in code are what matter for test fixtures that do not set the env var.

This PR adds no new test for the README; the README change is documentation only and does not need automated coverage. The validator unit tests are the regression net.
