# PR-024: Drives page redesign

Branch: pr-024-drives-page-redesign
- Type: ux
- Complexity: medium
- Depends on: PR-023
- Priority: 4
- Coder: any

## Problem

The current drives page renders the same physical-drive table that used to live on the dashboard. After PR-023 removes the table from the dashboard, the drives page becomes the canonical place for per-drive data, and it needs a redesign that matches the §3 design system: dense data table, sortable columns, status indicators on each row, slot id rendered as a link to drive detail.

The existing table is functional but lacks: column sorting, consistent status icon/badge per row, slot-as-link affordance, and the typography/density tokens from PR-018.

## Scope

In scope:
- Refactor `templates/pages/drives.html` and `templates/partials/physical_drive_table.html` to use the §3 DataTable component pattern.
- Sortable columns: slot, model, state, temperature, size. Sort arrows in the column headers; click toggles asc/desc.
- Each row's slot cell is a link to `/drives/<e>:<s>` (drive detail page already exists).
- Each row carries a status indicator: green/yellow/red dot or icon at row-start, derived from drive state + temperature thresholds.
- Header includes a small summary count: "12 drives, 11 optimal, 1 warning".
- Keep the existing data layer; sorting happens client-side on the rendered table for simplicity (the dataset is at most around 24 drives).
- Tests for the partial render, the sort state markup, and the slot-link generation.

Out of scope:
- Filter chips on drives. The drive page is not high-volume; filter is YAGNI.
- Pagination. Datasets stay small.
- Per-drive expand/collapse. Drive detail page exists; clicking the slot navigates.
- Real-time updates beyond the existing 30 s auto-refresh.

## Implementation notes

Client-side sort over server-side: server-side would require pushing sort params into the existing query, which currently returns a snapshot-shaped result already ordered by enclosure:slot. Client-side keeps the data layer untouched and works fine for around 24 rows. If the dataset ever scales, switch to server-side then.

The "dot or icon" indicator on each row uses the same Lucide sprite from PR-019. For optimal: `check-circle`. For warning: `alert-triangle`. For critical: `x-circle`. Re-using one sprite across the application keeps the page weight bounded.

Slot id format `<e>:<s>` (e.g. `2:0`) matches storcli vocabulary per §3.1 "recognition over recall". Operators familiar with storcli read this naturally; an operator new to the tool sees the same notation as `storcli /c0/e2/s0 show`.

The temperature cell color thresholds are taken from `Settings.temp_warning_celsius` / `temp_critical_celsius`, which are already used by the event detector. Reuse via a small helper `_temp_status(value, settings) -> Literal["optimal", "warning", "critical", "neutral"]` rather than re-deriving inline.

Sort persistence across reloads: out of scope. The operator picks a sort each visit. Storing in localStorage would survive the §3.5 "predictable interface" rule but adds complexity.
