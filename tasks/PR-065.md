## Problem

MegaRAID's patrol read scans all sectors of all drives during idle time, detecting bad sectors before they hit production read paths. By default it runs continuously at low priority; operators may want to schedule it (e.g. only during off-hours) or trigger an on-demand pass after suspecting an issue.

This PR exposes patrol-read controls: view current state, trigger manual run, set automatic schedule. All read-mostly; the manual trigger is the one mutating action.

## Scope

In scope:
- `GET /controller/patrol-read` returning current patrol-read state (mode, progress percent, last run timestamp).
- `POST /controller/patrol-read/start` triggers a manual patrol read.
- `POST /controller/patrol-read/stop` aborts an in-progress patrol.
- `POST /controller/patrol-read/mode` body `{mode: "auto" | "manual" | "disable"}`.
- Storcli runner whitelist extended.
- A small UI section on the controller-detail page (or overview footer if no controller-detail page exists yet).
- Audit events on every mutating endpoint.

Out of scope:
- Schedule by time-of-day (cron-style). The controller's auto mode picks idle time itself; replicating cron in the dashboard adds little.
- Per-drive patrol read. Whole-controller scope only.

## Implementation notes

Patrol read is non-destructive (read-only scan). Risk is operational: starting a patrol on a busy production array may impact performance for hours. The UI should warn before Start.

The audit message: `"patrol read start"`, `"patrol read stop"`, `"patrol read mode set to <X>"`. The events page filter for `operator_action` (PR-035) surfaces these.

The controller's auto mode is the right default; this PR exposes overrides for the rare cases where the operator wants explicit control.
