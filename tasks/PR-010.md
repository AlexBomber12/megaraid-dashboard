# PR-010: Dynamic local time via JS

Branch: pr-010-dynamic-local-time
- Type: ux
- Complexity: low
- Depends on: none
- Priority: 2
- Coder: any

## Problem

The dashboard currently renders timestamps in two formats: UTC and Europe/Rome (CEST/CET) via the `utc_to_cest` Jinja filter and parallel "captured at" labels. Hardcoding Europe/Rome is wrong for several reasons:

1. The operator may use the dashboard from outside Europe (Aleksei travels; the production NAS is in Italy but the browser is wherever the operator is). The Italian display time then mismatches the operator's wristwatch.
2. CEST and CET shift twice a year and the conversion logic must track that. Server-side is the wrong place for a per-user concern.
3. The existing UI shows two timestamps per event (UTC + CEST), which doubles screen real estate for an operator who only cares about one of them.

The roadmap principle (§2.3) says: server emits UTC ISO 8601, browser converts to local via JS. This PR implements that. Server-side timezone code stays where it is (still useful for log lines and emails); the UI uses JS to render `<time datetime="...">` elements in the browser's own timezone. A `<noscript>` fallback shows UTC explicitly so JS-disabled clients still get correct (if unlocalized) data.

## Scope

In scope:
- Update server templates so every operator-visible timestamp renders as `<time datetime="<ISO-8601-UTC>" data-local-time>...</time>` with the inner text being a UTC fallback (e.g. `2026-05-02 18:30 UTC`).
- A small JS file `static/js/local-time.js` (around 30 lines) that:
  - Selects all `time[data-local-time]` elements.
  - Replaces inner text with `Intl.DateTimeFormat` output in the browser's resolved locale and timezone.
  - Runs on `DOMContentLoaded` and on the htmx `htmx:afterSwap` event so partial-refreshed sections also localize.
- A header clock `<time class="utc-clock" data-local-time-clock>...</time>` updating every second to show current local time (replacing the existing UTC clock element).
- `<noscript>` fallback wrapper showing `UTC` explicitly so non-JS clients see unambiguous timestamps.
- A new Jinja helper or filter `iso_utc(datetime)` that returns `2026-05-02T18:30:00Z` (or equivalent strict ISO 8601 with `Z`), used by every template that emits a timestamp.
- Tests for: helper output format, template rendering produces `<time datetime>` markup, noscript wrapper present.

Out of scope:
- Removing the `utc_to_cest` filter immediately. Keep it for one release in case it is referenced by alert email templates or anywhere else; mark deprecated in code comments. A follow-up PR can delete it once all references are JS-driven.
- 12-hour vs 24-hour preference. The browser's `Intl.DateTimeFormat` defaults to the user's locale; respect it.
- A user-facing timezone picker or locale override. The browser's resolved zone is the right answer.
- Updating alert email content to use UTC. Email is read in mail clients without JS; the existing UTC + Europe/Rome dual rendering in emails is fine and out of scope.
- Server-side timezone removal or refactor. Server may still log in UTC + Europe/Rome.
- Updating event timestamp display formats beyond the template-level change (e.g. relative time like "5 minutes ago"). Out of scope; revisit in PR-025 or a polish PR.

## Files to create

`src/megaraid_dashboard/static/js/local-time.js`:
- Around 30 lines. No frameworks. Plain DOM API.
- Skeleton:

```js
(function () {
  function fmt(date, opts) {
    return new Intl.DateTimeFormat(undefined, opts).format(date);
  }

  function localizeAll(root) {
    const els = (root || document).querySelectorAll("time[data-local-time]");
    els.forEach(function (el) {
      const iso = el.getAttribute("datetime");
      if (!iso) return;
      const d = new Date(iso);
      if (isNaN(d.getTime())) return;
      const opts = el.dataset.localTimeFormat === "date"
        ? { year: "numeric", month: "short", day: "2-digit" }
        : el.dataset.localTimeFormat === "time"
          ? { hour: "2-digit", minute: "2-digit", second: "2-digit" }
          : { year: "numeric", month: "short", day: "2-digit",
              hour: "2-digit", minute: "2-digit" };
      el.textContent = fmt(d, opts);
      el.title = iso;
    });
  }

  function startClock() {
    const el = document.querySelector("[data-local-time-clock]");
    if (!el) return;
    function tick() {
      el.textContent = fmt(new Date(), {
        hour: "2-digit", minute: "2-digit", second: "2-digit"
      });
    }
    tick();
    setInterval(tick, 1000);
  }

  document.addEventListener("DOMContentLoaded", function () {
    localizeAll(document);
    startClock();
  });

  document.body && document.body.addEventListener("htmx:afterSwap", function (evt) {
    localizeAll(evt.target || document);
  });
})();
```

- Note: the htmx listener is attached inside the IIFE; if `document.body` is not yet available at script load, defer the binding into `DOMContentLoaded`. Adjust accordingly.

```python
def iso_utc(value: datetime | None) -> str:
    if value is None:
        return ""
    if value.tzinfo is None:
        raise ValueError("iso_utc requires a timezone-aware datetime")
    return value.astimezone(UTC).strftime("%Y-%m-%dT%H:%M:%SZ")
```

## Implementation notes

`Intl.DateTimeFormat(undefined, ...)` resolves to the browser's locale and timezone automatically. Do not pass an explicit timezone; that would re-introduce the hardcoding the PR is trying to remove. Test in a browser with locale set to `en-US` and again with `it-IT` to confirm the format adapts.

The `<noscript>` fallback explicitly says `UTC` (text suffix) so a JS-disabled operator reading the page knows the values are UTC and not their local time. Without the suffix, the operator would see numbers and assume local. The same rule applies to the inner text of `<time>` tags before JS replaces it: the inner text is the UTC string with `UTC` suffix, so even during the brief paint-before-JS-runs, the user is not lied to.

The htmx integration: htmx fires `htmx:afterSwap` on the swapped element after content is replaced. Listening on `document.body` catches all swaps. Inside the listener, run `localizeAll(evt.target)` so we only re-process the changed DOM subtree, not the whole page. This keeps the work bounded as the events page grows.

The header clock fires every second. Browser tabs in background throttle `setInterval` to once per minute, which is acceptable for a clock; do not work around it.

The `aria-live="off"` on the clock prevents screen readers from announcing the second tick. The clock is decorative; nobody needs to hear "12 colon 34 colon 56" out loud once a second.

The `iso_utc` filter requires timezone-aware datetimes by design. The codebase already uses `UTCDateTime` SQLAlchemy column type; reads from the DB return aware datetimes. If a test fixture passes a naive datetime, the `ValueError` surfaces the bug at the boundary instead of silently producing a string in the wrong zone.

The deprecation comment on `utc_to_cest` is a marker for a future cleanup PR. Do NOT delete the filter in this PR; alert email templates may still use it, and a partial migration that breaks emails is worse than carrying the filter for one more release.

If the existing templates use a different timestamp pattern (e.g. a macro in `_macros.html`), refactor to a new `local_time(ts)` macro and use it everywhere instead of inlining the noscript+time pair across many files. Cleaner, less error-prone, and one place to change later.
