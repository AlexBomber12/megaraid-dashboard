# PR-012: Alembic auto-upgrade in systemd

Branch: pr-012-alembic-preflight
- Type: config
- Complexity: low
- Depends on: none
- Priority: 3
- Coder: any

## Problem

`src/megaraid_dashboard/app.py` calls `_upgrade_database` inside the FastAPI lifespan, which runs Alembic upgrade head at process start. That works for the running process, but couples DB migration to the web app boot. Two failure modes follow:
1. If the migration fails mid-way (e.g. permission denied on the SQLite WAL file), the web app errors at startup and systemd restarts it in a loop, hiding the root cause behind generic restart-loop noise.
2. There is no externally observable preflight: a fresh deploy first runs the web app, which runs the migration; the operator cannot verify migration before flipping traffic.

The fix is a `scripts/preflight.sh` that runs `alembic upgrade head`, validates DB write access, and exits non-zero on failure, plus a sample systemd unit fragment that wires `ExecStartPre=` to it. systemd surfaces preflight failures distinctly from app failures and prevents the web app from starting when the DB is in a known-bad state. INSTALL.md (PR-056) will reference this script.

## Scope

In scope:
- `scripts/preflight.sh`: bash script that runs `alembic upgrade head` and verifies DB writability via a temp-table create-and-drop.
- `deploy/megaraid-dashboard.service` sample systemd unit including `ExecStartPre=/opt/megaraid-dashboard/scripts/preflight.sh`.
- README mention pointing at the preflight script.
- A small change in `app.py` lifespan: keep the in-process `_upgrade_database` call as a safety net (idempotent), but downgrade its log level to debug when the DB is already at head, so production logs are not noisy.
- Tests for the preflight script (bash-level: stub `alembic`, run script, assert exit codes and output).

Out of scope:
- Removing the in-process migration call. Belt and suspenders are fine; removing means a fresh dev environment without preflight would not auto-migrate. Keep it.
- A migration-rollback script. Operator runs `alembic downgrade -1` manually if needed.
- Locking around concurrent `alembic upgrade` calls. SQLite serializes via the WAL; concurrent calls collapse to a no-op for the second.
- Making the systemd unit production-final. The repo carries it as a sample; the actual installed unit is generated by `scripts/install.sh` (PR-031).

## Files to create

`scripts/preflight.sh`:
```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

echo "==> alembic upgrade head"
.venv/bin/alembic upgrade head

echo "==> DB writability probe"
.venv/bin/python - <<'PY'
import os, sqlite3, sys, tempfile
from urllib.parse import urlparse, unquote

url = os.environ.get("DATABASE_URL", "sqlite:///./megaraid.db")
if not url.startswith("sqlite:"):
    sys.exit(0)  # non-sqlite: skip; assume preflight enough
parsed = urlparse(url)
path = unquote(parsed.path.lstrip("/")) or parsed.netloc
conn = sqlite3.connect(path, timeout=2)
try:
    conn.execute("CREATE TABLE IF NOT EXISTS _preflight (n INT)")
    conn.execute("INSERT INTO _preflight VALUES (1)")
    conn.execute("DROP TABLE _preflight")
    conn.commit()
finally:
    conn.close()
print("DB writable")
PY

echo "==> preflight OK"
```
- Mode 0755 (`chmod +x`).
- Loads `DATABASE_URL` from environment; the systemd unit's `EnvironmentFile=` puts `.env` values into env.

`deploy/megaraid-dashboard.service`:
```
[Unit]
Description=MegaRAID Dashboard
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=megaraid
Group=megaraid
WorkingDirectory=/opt/megaraid-dashboard
EnvironmentFile=/etc/megaraid-dashboard/env
ExecStartPre=/opt/megaraid-dashboard/scripts/preflight.sh
ExecStart=/opt/megaraid-dashboard/.venv/bin/uvicorn megaraid_dashboard.app:create_app --factory --host 0.0.0.0 --port 8090
Restart=on-failure
RestartSec=5

NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/megaraid-dashboard

[Install]
WantedBy=multi-user.target
```
- Sample only. PR-031 generates the live one.

## Implementation notes

`set -euo pipefail` is mandatory. Without it, a failure inside the heredoc would be swallowed and systemd would proceed to start the web app against a broken DB.

The DB writability probe creates and drops a `_preflight` table rather than just opening the connection. Connection open succeeds against a read-only file; the create/insert/drop sequence catches that.

The systemd hardening flags (`NoNewPrivileges`, `ProtectSystem=strict`, `ReadWritePaths`) are belt-and-suspenders. PR-031 will revisit these alongside the sudoers fragment for storcli.

The `--factory` flag on uvicorn is required because `create_app()` is a callable that returns the FastAPI instance. Without `--factory`, uvicorn imports `create_app` itself as the app, which is wrong.

The downgrade of the log level for "already at head" is small but worth it: every restart currently logs `database_at_head_revision` at info, which fills up the journald log over months.

Edge case: if `DATABASE_URL` is absent from env, the script falls back to the same default the app uses (`sqlite:///./megaraid.db`). Document this in a comment so the operator knows it.
