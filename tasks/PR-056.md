## Problem

Per roadmap §10, INSTALL.md is a P0-priority document deferred to project end. The system is one-author-locked until it exists. This PR ships it.

The audience is an operator deploying from a fresh Ubuntu 24.04 install, who has access to the storcli binary and an SMTP credential, and who knows basic Linux administration but has never touched megaraid-dashboard.

## Scope

In scope:
- `INSTALL.md` covering:
  1. Pre-flight (5 prerequisites with verification commands).
  2. Step 1: Clone repository.
  3. Step 2: Run `scripts/install.sh` (wraps PR-028..PR-031).
  4. Step 3: Edit `.env` if config wizard skipped.
  5. Step 4: Verify `systemctl status megaraid-dashboard.service`.
  6. Step 5: Verify health endpoint (`curl localhost:8090/healthz`).
  7. Step 6: Configure nginx reverse proxy (link to `docs/PROXY-SETUP.md`).
  8. Step 7: First login, change admin password, configure SMTP test send.
  9. Smoke test checklist (10 items).
  10. Common pitfalls cross-reference to TROUBLESHOOTING.md (PR-058).
- Around 400 lines markdown.

Out of scope:
- RUNBOOK contents. PR-057.
- TROUBLESHOOTING contents. PR-058.
- Architecture deep-dive. PR-057.
- Multi-host deploy. Single host per §2.7.

## Implementation notes

INSTALL.md is the contract for §2.8: "A new operator deploys from INSTALL.md alone." Validate the document by having someone unfamiliar follow it on a fresh Ubuntu 24.04 VM. The success criterion is "60 minutes from clone to a working dashboard" per §8.

Linear flow over branched flow: branched docs ("if you have X, do Y; if not, do Z") create stress and increase the chance of an operator missing a branch. Where two paths exist, document the more common one and put alternatives in TROUBLESHOOTING.

Cross-references rather than inlining: linking to `docs/PROXY-SETUP.md` for nginx instead of duplicating keeps INSTALL.md focused on "first-time deploy" and PROXY-SETUP focused on "proxy specifics". A future change to nginx configuration touches one file.

The 10-item smoke test at the end is the operator's confirmation that everything works. Each item is a single command with a single expected output. If any item fails, the operator knows the deploy is incomplete before declaring done.
