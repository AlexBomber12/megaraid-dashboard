## Problem

PR-040 ships the storage. This PR wires it: two operator endpoints (start / stop), notifier respect for the window, audit events on transitions, automatic expiration handling.

## Scope

In scope:
- `POST /maintenance/start` body `{duration_minutes: int, reason: str}`. Sets state `active=True`, computes `expires_at=now+duration`, records `started_by=auth_username`. Audit event recorded.
- `POST /maintenance/stop` clears state. Audit event recorded.
- `GET /maintenance` returns current state JSON: `active`, `expires_at`, `started_by`, `remaining_seconds`.
- Notifier integration: in the loop that sends emails, check `get_maintenance_state(session, now=...)`. If active, skip sending. The notifier still records that it WOULD have sent (a structlog line: `notifier_skipped_for_maintenance`) so debugging is easy.
- Validation: `duration_minutes` must be in `[1, 1440]` (between 1 minute and 24 hours). Reason is required, non-empty, max 200 chars.
- Tests: endpoint validation, state transitions, audit emission, notifier skip behavior.

Out of scope:
- UI banner. PR-042.
- Stop-on-stop idempotency edge case. Stop while inactive is a no-op (200, no audit event).
- Pause/resume of the collector. §7 explicitly keeps the collector recording during maintenance.

## Files to modify

`src/megaraid_dashboard/web/routes.py`:
- Add three routes:

```python
class MaintenanceStartRequest(BaseModel):
    duration_minutes: int = Field(ge=1, le=1440)
    reason: str = Field(min_length=1, max_length=200)


@router.post("/maintenance/start")
async def maintenance_start(
    body: MaintenanceStartRequest, request: Request
) -> JSONResponse:
    now = datetime.now(UTC)
    expires = now + timedelta(minutes=body.duration_minutes)
    username = request.scope.get("user_username", "unknown")
    with request.app.state.session_factory() as session, session.begin():
        set_maintenance_state(
            session, active=True, expires_at=expires, started_by=username,
        )
        record_operator_action(
            session, username=username,
            message=f"maintenance start duration {body.duration_minutes} min reason: {body.reason}",
        )
    return JSONResponse({
        "active": True,
        "expires_at": expires.isoformat(),
        "started_by": username,
    })


@router.post("/maintenance/stop")
async def maintenance_stop(request: Request) -> JSONResponse:
    now = datetime.now(UTC)
    username = request.scope.get("user_username", "unknown")
    with request.app.state.session_factory() as session, session.begin():
        prior = get_maintenance_state(session, now=now)
        set_maintenance_state(session, active=False, expires_at=None, started_by=None)
        if prior.active:
            record_operator_action(
                session, username=username,
                message="maintenance stop",
            )
    return JSONResponse({"active": False})


@router.get("/maintenance")
async def maintenance_get(request: Request) -> JSONResponse:
    now = datetime.now(UTC)
    with request.app.state.session_factory() as session:
        state = get_maintenance_state(session, now=now)
    remaining = (state.expires_at - now).total_seconds() if state.active and state.expires_at else None
    return JSONResponse({
        "active": state.active,
        "expires_at": state.expires_at.isoformat() if state.expires_at else None,
        "started_by": state.started_by,
        "remaining_seconds": int(remaining) if remaining and remaining > 0 else None,
    })
```

## Implementation notes

The notifier check happens INSIDE the send loop, not in the scheduler. The scheduler keeps firing every 60 seconds; the notifier just no-ops while maintenance is active. This way, when maintenance ends, the very next tick sees pending events from the maintenance window and sends them (subject to suppression / throttle rules already in place).

Actually that last point matters for the operator: events accumulated DURING maintenance fire as a batch the moment maintenance ends. If the array was loud (frequent rebuild progress changes, brief offline blips), the email inbox gets a flood. The existing `alert_suppress_window_minutes` and `alert_throttle_per_hour` from `Settings` cap that flood, but document the behavior in the operator-facing message: "maintenance mode pauses notifications; queued events flush on stop".

The 1440-minute (24-hour) cap on duration is conservative. A maintenance window longer than a day is unusual and likely indicates the operator forgot to stop it. Force a deliberate restart in that case.

The reason field is recorded in the audit log only; it does not appear elsewhere in the UI for now (PR-042 may surface it on the banner). It exists for forensic traceability: "why was maintenance active from 14:00 to 14:30 on 2026-05-10".

The structlog `notifier_skipped_for_maintenance` line is the operational signal. Without it, debugging "why did I not get an alert?" requires inspecting the DB; with it, `journalctl` shows the answer.

Audit message format: `"maintenance start duration N min reason: <text>"`. PR-035 audit page renders this verbatim. If a future PR wants structured display, parse from this format.
