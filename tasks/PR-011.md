# PR-011: Health endpoint

Branch: pr-011-health-endpoint
- Type: feature
- Complexity: low
- Depends on: PR-005
- Priority: 3
- Coder: any

## Problem

The dashboard has no machine-readable liveness signal. systemd `ExecStartPre`, the future Prometheus blackbox probe (PR-043), nginx upstream health checks (PR-013), and any uptime monitor cannot tell "process is running" from "process is hung in a database deadlock". A trivial root path GET returns the HTML overview page (around 200 KB and several DB queries), which is too expensive and too brittle to use as a probe.

The PR-005 auth middleware already whitelists `/healthz`, so adding the route here completes the contract.

## Scope

In scope:
- `GET /healthz` returning JSON with three keys: `status` (string `ok` or `degraded`), `database` (string `ok` or `error`), `collector` (string `ok`, `idle`, or `lock_held`).
- 200 when `status=ok`, 503 when `status=degraded`.
- Sub-100 ms latency: a single `SELECT 1` against the DB plus a check of `app.state.collector` and `app.state.collector_lock_fd`.
- No HTML rendering, no template engine.
- Whitelisted from auth (already is, via PR-005).
- Tests covering: healthy, DB unreachable, collector disabled (idle), collector lock held by another instance.

Out of scope:
- Liveness vs readiness split. One endpoint is sufficient; if Prometheus or k8s style split is needed later, add `/readyz` then.
- Detailed component breakdown (per-table, per-service). The three keys above are enough for a homelab.
- Authentication on the endpoint. The whole point is unauth probing.
- Metrics emission. That is PR-043.

## Implementation notes

The "degraded" classification is conservative: only DB error promotes status to degraded. Collector states `idle` and `lock_held` both stay `ok` because:
- Idle (`collector_enabled=False`) is operator-chosen via env var, not a failure.
- Lock held means a sibling instance is collecting; from this process's perspective, the system is fine.

The `SELECT 1` should run with `engine.connect().execution_options(isolation_level="AUTOCOMMIT")` and a connection-level timeout (e.g. `connect_args={"timeout": 2}` for SQLite) so a wedged DB does not hang the probe.

The 503 status code (not 500) tells nginx upstream-health logic to mark the backend as down; 500 would still be considered "alive but error". For uptime checks specifically, 503 is the right code.

`Cache-Control: no-store` prevents intermediate caches (nginx, browser) from serving a stale 200 after the backend went down.

## Success criteria

1. `python -m ruff check .`, `ruff format --check .`, `mypy src`, `pytest` all pass.
2. `curl -i http://localhost:8090/healthz` returns 200 with the three-key JSON when the system is healthy.
3. With the DB connection broken (e.g. invalid `database_url`), the same call returns 503 and `database: error`.
4. With `collector_enabled=False`, the call returns 200 and `collector: idle`.
